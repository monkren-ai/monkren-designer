# 方法层 · 创作与改进

> 当前活跃创作链：需求澄清 → 线框或变体 → 可交互原型 → 审查 → 授权后改进。具体执行优先遵循相应 `skills/` 模块。

## 1. 权限与输入

- “建议、分析、可以怎么做”默认只输出方案。
- “制作、生成、实现、修复、改进”允许在用户给定范围内修改文件。
- 先读取项目已有设计系统、组件、路由、品牌资产和技术栈；不要为了套模板替换现有上下文。
- 不知道的业务数据、客户、指标和品牌细节必须标成 placeholder。

开始前确认最少信息：目标用户、核心任务、目标平台、现有上下文、不可妥协的约束。已能从项目或用户消息确定的内容不要重复询问。

## 2. 选择最便宜的有效产出

| 不确定性 | 适合的产出 | 模块 |
|---|---|---|
| 信息架构、页面骨架、步骤数 | 灰度线框 | `skills/02-create/wireframe/SKILL.md` |
| 视觉语言、布局方向、交互模型 | 3 个以上变体 | `skills/02-create/generate-variations/SKILL.md` |
| 流程、状态、校验、可用性 | 可交互原型 | `skills/03-execute/make-a-prototype/SKILL.md` |
| 已有报告中的确定问题 | 修复与复验 | `skills/05-improve/design-improve/SKILL.md` |

不要在结构尚未确定时投入高保真；也不要在线框已经回答问题后继续装饰它。

## 3. 线框方法

1. 明确探索对象、用户目标、平台和 1–2 条变化轴。
2. 使用黑、白、灰、系统字体、标注方框和诚实占位图。
3. 至少生成 3 个结构上不同的方向；一个稳妥、一个精炼、一个偏离常规。
4. 每个方向附 2–4 条取舍说明。
5. 记录用户选择、拒绝原因和新约束，作为高保真 brief。

默认用一个自包含 HTML/SVG 文件并排展示。简单矩形和连接线可以使用 SVG；不要在低保真阶段绘制装饰性插画。

## 4. 变体方法

选择 2–4 条有意义的变化轴：布局、信息层级、密度、交互模型、字体系统、色彩策略或语气。变体必须在这些轴上实质不同，不只是换按钮颜色。

推荐顺序：

1. **稳妥**：遵守已有模式。
2. **精炼**：保留结构，强化一两个维度。
3. **新颖**：提出真正不同且仍服务目标的方案。

每个方向写一句可区分的描述，并给出有理由的推荐。默认一个文件承载全部方向；不要散落成 `v1.html`、`v2.html`、`v3.html`。

## 5. 原型方法

### 5.1 先画状态图

列出入口、屏幕、主要动作、成功、失败、返回路径和持久化需求。一个原型至少应测试一条完整核心流程，而不是只让静态页面互相跳转。

### 5.2 实现原则

- 项目已有框架时沿用其组件与构建方式；独立演示默认使用自包含 HTML/CSS/JavaScript。
- 接通表单校验、错误清除、成功反馈、返回路径和可见状态变化。
- 只有异步动作需要 loading 与防重复提交。
- 仅持久化非敏感且有演示价值的状态；邮箱、电话、凭据默认只留在内存。
- 使用语义 HTML、可见 focus、键盘导航和 reduced-motion。

### 5.3 验证

逐条走完整流程：CTA 有去处、表单能失败也能恢复、状态切换可见、Tab/Enter/Escape 合理、320px 到桌面宽度无核心内容裁切。无法运行的行为必须明确标注“未验证”。

## 6. 基于报告改进

1. 解析报告中的位置、证据、严重度、建议和依赖。
2. 先复核问题仍然存在，避免依据过期报告修改。
3. 按 P0 → P1 → P2 → P3 执行；方向性争议先保留为未决项。
4. 每次修改保持项目设计系统和技术结构，不做无关重写。
5. 运行与风险相称的验证，并把每项修复映射回原 finding。

如果修复暴露新需求，回到调研；如果方向本身错误，回到线框或变体；如果只是实现质量问题，修复后回到审查。

## 7. 媒介纪律

- CSS 负责布局、排版、响应式和状态；SVG 负责图标、图表和简单几何，不承担复杂插画。
- 动效用于解释状态变化，不用于填充空洞体验。
- 真实图片优先使用用户或品牌提供的资产；缺失时使用明确占位。
- 固定尺寸画布在小屏应缩放或重排，不能被裁切。
- 所有颜色、间距和字号尽量追溯到已有 token。

## 8. 交付

交付摘要只需说明：生成或修改的文件、已验证的行为、使用的模拟内容、未验证项和仍需用户决定的事项。不要重复整份设计说明，也不要声称没有实际运行过的测试已经通过。
