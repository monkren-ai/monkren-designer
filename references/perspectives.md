# 多视角审查体系

> 10 维度审查框架、场景权重配置、S/W/F 输出格式。作为设计审查的系统化基线，确保不同审查者对同一产出有一致的判断标准。

---

## 1. 视觉审美维度

### 评审标准

#### 色彩搭配

| 维度 | 评审要点 | 评分 |
|------|----------|------|
| 和谐度 | 色彩是否和谐统一（色相/明度/饱和度关系） | 1-10 |
| 对比度 | 前景与背景对比是否足够（正文 ≥ 4.5:1，大文本 ≥ 3:1） | 1-10 |
| 品牌色应用 | 品牌色是否正确使用，比例是否恰当 | 1-10 |

#### 字体选择

| 维度 | 评审要点 | 评分 |
|------|----------|------|
| 可读性 | 字体在目标尺寸和载体上是否清晰可读 | 1-10 |
| 匹配度 | 字体风格是否与设计哲学/品牌调性匹配 | 1-10 |
| 平台惯例 | 字体选择是否符合平台规范（iOS San Francisco/Android Roboto/Web 安全字体） | 1-10 |

#### 空间布局

| 维度 | 评审要点 | 评分 |
|------|----------|------|
| 呼吸感 | 留白比例是否合理（通常 ≥ 40%），信息是否拥挤 | 1-10 |
| 密度 | 信息密度与载体是否匹配（PPT 稀、PDF 密） | 1-10 |
| 平衡感 | 元素分布是否均衡，有无一边过重一边过轻 | 1-10 |

#### 视觉层次

| 维度 | 评审要点 | 评分 |
|------|----------|------|
| 焦点明确 | 是否有明确的视觉焦点引导用户视线 | 1-10 |
| 信息分层 | 标题/正文/注释等层级是否清晰（≥ 2.5 倍字号对比） | 1-10 |
| 引导流 | 用户视线是否沿设计者意图自然流动 | 1-10 |

### S/W/F 输出格式

```markdown
### 视觉审美

**Strength**
- [维度] 具体优点（引用具体元素/色值/字号作为证据）

**Weakness**
- [维度] 具体问题（引用具体元素/色值/字号作为证据）

**Fix**
- [维度] 修复建议（具体可操作的改进方案）
```

**示例**：

```markdown
### 视觉审美

**Strength**
- 色彩：品牌色 #D97757 应用比例恰当，主色占比约 60%，辅色点缀不过度
- 字体：Instrument Serif + Geist Sans 配对有特色，与品牌调性匹配度高

**Weakness**
- 色彩：正文 #1A1A1A 与背景 #FFFFFF 对比度 16:1 过高，长时间阅读易疲劳
- 布局：Section B 留白占比仅 25%，信息密度过高，与 Section A 的 50% 留白失衡
- 层级：副标题 20px 与正文 16px 字号差距 1.25 倍，中间层级模糊

**Fix**
- 色彩：正文色调整为 #3D3D3D（对比度约 10:1），更符合 WCAG 舒适阅读标准
- 布局：Section B 增加 padding 或减少内容，恢复 40%+ 留白比例
- 层级：副标题调整为 24px，与正文形成 1.5 倍对比，明确中间层级
```

## 2. 交互体验维度

> 适用于 App / Web / 桌面端。不同端型评审侧重点有所不同。

### 评审标准

#### 操作流程合理性

| 维度 | App | Web | 桌面端 |
|------|-----|-----|--------|
| 路径长度 | 核心操作是否 ≤ 3 步 | 核心操作是否 ≤ 3 次点击 | 核心操作是否 ≤ 2 次操作 |
| 步骤数 | 单个任务步骤是否精简 | 表单步骤是否合理拆分 | 复杂任务是否支持批量操作 |
| 效率 | 操作路径是否符合用户 mental model | 导航深度是否合理（≤ 3 级） | 快捷键是否覆盖高频操作 |

#### 反馈机制完整性

| 维度 | 评审要点 | 评分 |
|------|----------|------|
| 视觉反馈 | 操作是否有即时的视觉响应（hover/active/loading 状态） | 1-10 |
| 触觉反馈 | （仅 App）按钮点击是否有触感反馈 | 1-10 |
| 文字提示 | 操作结果是否有明确的文字说明 | 1-10 |

#### 错误处理得当性

| 维度 | 评审要点 | 评分 |
|------|----------|------|
| 错误预防 | 是否通过约束条件预防用户输入错误 | 1-10 |
| 错误恢复 | 操作错误后是否提供简便的恢复路径 | 1-10 |
| 错误提示质量 | 错误提示是否具体、友好、有建设性 | 1-10 |

#### 学习成本

| 维度 | 评审要点 | 评分 |
|------|----------|------|
| 直觉性 | 操作方式是否符合用户直觉和习惯 | 1-10 |
| 一致性 | 同类操作是否有一致的交互模式 | 1-10 |
| 可发现性 | 功能是否容易被用户发现（引导/提示/默认值） | 1-10 |

### S/W/F 输出格式

```markdown
### 交互体验

**Strength**
- [维度] 具体优点（引用具体交互/组件/流程作为证据）

**Weakness**
- [维度] 具体问题（引用具体交互/组件/流程作为证据）

**Fix**
- [维度] 修复建议（具体可操作的改进方案）
```

**示例**：

```markdown
### 交互体验

**Strength**
- 操作流程：注册流程仅需 3 步（手机号→验证码→设置密码），路径长度合理
- 反馈机制：按钮 hover 状态有背景色变化，active 有 scale(0.98) 按压反馈
- 一致性：全局使用 PrimaryButton / SecondaryButton 组件，交互模式统一

**Weakness**
- 错误处理：表单错误仅在提交后显示红字，无实时校验（如邮箱格式错误应在输入时提示）
- 错误恢复：删除操作无确认弹窗，误删后无恢复入口
- 可发现性：高级设置隐藏在三级菜单，无引导提示，新用户难以发现

**Fix**
- 错误处理：添加实时校验，在输入框 blur 时触发格式校验，错误时显示具体提示（如"请输入有效的邮箱地址"）
- 错误恢复：删除操作添加确认 Dialog，提供"撤销"功能（5 秒内可撤回）
- 可发现性：高级设置添加"NEW"标签或首次使用引导 tooltip，降低发现门槛
```

## 3. 品牌一致性维度

### 评审标准

#### 品牌色应用

| 维度 | 评审要点 | 评分 |
|------|----------|------|
| 一致性 | 品牌色是否在各页面、组件、状态间保持一致 | 1-10 |
| 正确性 | 品牌色值是否与官方规范一致（非近似值） | 1-10 |
| 比例 | 品牌色使用比例是否恰当（主色 60-70%/辅色 20-30%/强调色 < 10%） | 1-10 |

#### 品牌语调

| 维度 | 评审要点 | 评分 |
|------|----------|------|
| 文字风格 | 文案语气/人称/词汇是否与品牌调性一致 | 1-10 |
| 视觉风格 | 整体视觉呈现是否传达品牌个性（简约/活泼/专业/权威等） | 1-10 |

#### 品牌资产使用规范

| 维度 | 评审要点 | 评分 |
|------|----------|------|
| Logo 使用 | Logo 是否使用真实文件，尺寸/位置/间距是否符合规范 | 1-10 |
| 图片风格 | 图片色调/风格是否与品牌调性统一 | 1-10 |
| 图标风格 | 图标风格（filled/outline/线重）是否统一，是否使用统一图标库 | 1-10 |

#### 品牌识别度

| 维度 | 评审要点 | 评分 |
|------|----------|------|
| 独特性 | 设计是否具有品牌独特性，而非通用模板 | 1-10 |
| 记忆度 | 用户能否在 3 秒后回忆出品牌视觉特征 | 1-10 |
| 情感连接 | 设计是否引发目标用户的情感共鸣 | 1-10 |

### S/W/F 输出格式

```markdown
### 品牌一致性

**Strength**
- [维度] 具体优点（引用具体元素/色值/文案作为证据）

**Weakness**
- [维度] 具体问题（引用具体元素/色值/文案作为证据）

**Fix**
- [维度] 修复建议（具体可操作的改进方案）
```

**示例**：

```markdown
### 品牌一致性

**Strength**
- 品牌色：主色 #D97757 贯穿所有页面和组件，按钮/链接/强调元素均使用品牌色
- 语调：文案使用"我们"第一人称，语气亲切友好，与品牌"温暖陪伴"调性一致
- 图标：全文统一使用 Lucide Icons，线条粗细一致，风格统一

**Weakness**
- Logo 使用：Logo 文件为 CSS 绘制（非真实 SVG 文件），与品牌真实 Logo 存在细节差异
- 图片风格：Hero 图使用冷色调照片，与品牌温暖调性冲突
- 识别度：去掉 Logo 后，设计与通用 SaaS 模板无明显差异，品牌独特性不足

**Fix**
- Logo 使用：联系品牌方获取真实 Logo 文件（SVG/PNG），替换当前 CSS 绘制版本
- 图片风格：更换为暖色调（橙/米色系）产品/场景照片，或使用品牌色滤镜处理现有图片
- 识别度：提取品牌独特视觉元素（如特定的图形符号/插画风格/纹理）应用于设计
```

## 4. 可访问性维度

### 评审标准

#### WCAG 2.1 AA 级核心要求

| 检查项 | 标准 | 检测方法 |
|--------|------|----------|
| 对比度 | 正文 vs 背景 ≥ 4.5:1，大文本（≥ 18pt 或 14pt bold）≥ 3:1 | Chrome DevTools Accessibility 面板 / Figma Contrast 插件 |
| 替代文本 | 图片有 alt 属性，图标按钮有 aria-label | 代码审查 |
| 键盘导航 | 焦点可见（focus ring），顺序合理（符合阅读顺序），快捷键不冲突 | 手动测试（Tab / Shift+Tab） |
| 屏幕阅读器 | 语义化标签（button/header/nav），aria 属性正确，阅读顺序合理 | NVDA/VoiceOver 测试 |

### 详细检查项

#### 对比度

| 检查点 | 标准 | 严重度 |
|--------|------|--------|
| 正文文字 vs 背景 | ≥ 4.5:1 | P0 |
| 大文本（≥ 18pt）vs 背景 | ≥ 3:1 | P0 |
| UI 组件边界（输入框/按钮）vs 相邻背景 | ≥ 3:1 | P0 |
| 装饰性内容（纯装饰图片） | 允许低对比度，但应有 aria-hidden | P2 |

#### 替代文本

| 检查点 | 标准 | 严重度 |
|--------|------|--------|
| 信息图片 |必须有 alt，描述图片内容而非"图片" | P0 |
| 功能图片（按钮内的图标）| aria-label 或 alt 描述功能 | P0 |
| 装饰性图片 | aria-hidden="true" | P2 |
| 图标按钮 | aria-label 描述功能 | P0 |

#### 键盘导航

| 检查点 | 标准 | 严重度 |
|--------|------|--------|
| 焦点可见 | focus ring 样式明确，颜色与背景对比 ≥ 3:1 | P0 |
| 焦点顺序 | 符合视觉/逻辑阅读顺序 | P0 |
| 快捷键 | 不与浏览器/辅助技术快捷键冲突 | P1 |
| 模态框 | 焦点锁定在模态框内，关闭后焦点返回触发元素 | P1 |
| 跳过链接 | 长页面提供"跳转到主要内容"链接 | P2 |

#### 屏幕阅读器兼容性

| 检查点 | 标准 | 严重度 |
|--------|------|--------|
| 语义化标签 | 使用 button/header/nav/article 等原生标签 | P0 |
| aria-label | 非语义元素（div/span）有 aria-label | P0 |
| aria-live | 动态内容更新区域有 aria-live 属性 | P1 |
| 阅读顺序 | DOM 顺序与视觉顺序一致 | P1 |

### S/W/F 输出格式

```markdown
### 可访问性

**Strength**
- [维度] 具体优点（引用具体元素/代码作为证据）

**Weakness**
- [维度] 具体问题（引用具体元素/代码作为证据）

**Fix**
- [维度] 修复建议（具体可操作的改进方案）
```

**示例**：

```markdown
### 可访问性

**Strength**
- 对比度：正文 #3D3D3D vs #FFFFFF 对比度 10:1，符合 WCAG AA 标准
- 替代文本：产品图片均包含描述性 alt 属性（如"XX产品界面截图"）
- 键盘导航：焦点顺序符合阅读顺序，focus ring 样式清晰（2px solid #D97757）

**Weakness**
- 对比度：次要文字 #888888 vs #FFFFFF 对比度 3.2:1，未达 4.5:1 标准（L45）
- 替代文本：搜索图标按钮无 aria-label，屏幕阅读器朗读为空白（L67）
- 键盘导航：模态框关闭后焦点未返回触发元素，用户需从页面顶部重新导航（L123）
- 屏幕阅读器：轮播图区域无 aria-live 属性，动态切换内容无法被屏幕阅读器感知（L156）

**Fix**
- 对比度：次要文字颜色调整为 #666666（对比度 5.9:1）
- 替代文本：搜索图标按钮添加 aria-label="搜索"
- 键盘导航：模态框关闭时使用 .focus() 将焦点返回触发元素（triggerButton.focus()）
- 屏幕阅读器：轮播图区域添加 aria-live="polite" 和 aria-atomic="true"
```

## 5. 商业转化维度

> 适用于营销页 / 落地页 / PPT。不同类型评审侧重点有所不同。

### 评审标准

#### 目标受众匹配度

| 维度 | 评审要点 | 评分 |
|------|----------|------|
| 用户画像 | 设计风格/语言是否匹配目标用户特征（年龄/职业/审美偏好） | 1-10 |
| 使用场景 | 设计是否考虑目标用户的实际使用场景（设备/环境/时间压力） | 1-10 |
| 需求痛点 | 文案/视觉是否直击目标用户的关键需求和痛点 | 1-10 |

#### 转化路径清晰度

| 维度 | 评审要点 | 评分 |
|------|----------|------|
| CTA 明显性 | CTA 是否足够醒目（颜色/尺寸/位置），是否在视觉焦点区 | 1-10 |
| 路径简洁性 | 从进入页面到完成转化是否路径最短、摩擦最小 | 1-10 |
| 摩擦点移除 | 是否移除了不必要的填写项/确认步骤/跳转 | 1-10 |

#### 价值主张传达

| 维度 | 评审要点 | 评分 |
|------|----------|------|
| 清晰度 | 核心价值主张是否在 3 秒内可被理解 | 1-10 |
| 差异化 | 是否明确传达与竞品的差异点 | 1-10 |
| 可信度 | 价值主张是否有数据/案例/背书支撑 | 1-10 |

#### 信任元素布局

| 维度 | 评审要点 | 评分 |
|------|----------|------|
| 社会证明 | 是否展示用户评价/案例/知名客户 logo | 1-10 |
| 资质展示 | 是否展示资质证书/行业奖项/媒体报道 | 1-10 |
| 安全保障 | 是否传达数据安全/隐私保护/退款保障 | 1-10 |

### S/W/F 输出格式

```markdown
### 商业转化

**Strength**
- [维度] 具体优点（引用具体元素/文案/数据作为证据）

**Weakness**
- [维度] 具体问题（引用具体元素/文案/数据作为证据）

**Fix**
- [维度] 修复建议（具体可操作的改进方案）
```

**示例**：

```markdown
### 商业转化

**Strength**
- CTA 明显性：主 CTA 使用品牌色背景 + 白字 + 阴影，尺寸 48px 高度，在首屏视觉焦点区
- 价值主张：Hero 标题"减少 40% 审批时间"具体可量化，与竞品泛泛的"提效赋能"形成差异
- 社会证明：展示 3 个真实客户 logo + 对应评价，评价包含具体数据（而非泛泛好评）

**Weakness**
- 转化路径：注册表单要求填写 8 个字段（含公司规模/职位/行业），增加用户决策摩擦
- 信任元素：金融类产品缺少安全认证 logo（如 SSL/PCI DSS），用户无法快速建立信任
- 需求痛点：文案聚焦功能而非用户收益，"支持 200+ 集成"不如"减少 50% 切换工具的时间"

**Fix**
- 转化路径：注册表单精简为 3 个必填字段（邮箱/密码/手机），其余字段延后至引导流程补充
- 信任元素：在页脚区域添加安全认证 logo 合集（SSL / 隐私保护 / 退款保障），降低用户决策风险
- 需求痛点：功能描述改为收益导向（"自动同步" → "每次更新自动同步，节省每天 30 分钟手动操作"）
```

## 6. 用户体验维度

> 适用于 App / Web / 桌面端。关注用户宏观感受，从认知负荷、情感设计、心智模型、信息架构、一致性体验等层面评估整体体验质量。

### 评审标准

#### 认知负荷

| 维度 | 评审要点 | 评分 |
|------|----------|------|
| 信息量 | 信息量是否超出工作记忆容量（7±2 原则） | 1-10 |
| 多源处理 | 是否需要同时处理多个信息源 | 1-10 |
| 决策点 | 决策点是否过多（选择悖论） | 1-10 |
| 学习成本 | 是否利用已有认知降低学习成本 | 1-10 |

#### 情感设计

| 维度 | 评审要点 | 评分 |
|------|----------|------|
| 本能层 | 视觉吸引力/第一印象是否积极 | 1-10 |
| 行为层 | 操作流畅感/成就感是否满足 | 1-10 |
| 反思层 | 品牌认同/自我表达是否到位 | 1-10 |
| 愉悦感 | 是否有愉悦感微交互 | 1-10 |
| 负面触发 | 是否避免负面情感触发 | 1-10 |

#### 心智模型匹配

| 维度 | 评审要点 | 评分 |
|------|----------|------|
| 概念一致性 | 概念模型是否与用户心智模型一致 | 1-10 |
| 隐喻恰当性 | 隐喻是否恰当 | 1-10 |
| 习惯尊重 | 是否尊重用户习惯 | 1-10 |
| 预期一致性 | 用户预期与实际行为是否一致 | 1-10 |

#### 信息架构

| 维度 | 评审要点 | 评分 |
|------|----------|------|
| 分类合理性 | 信息分类是否符合用户思维 | 1-10 |

| 命名一致 | 同一功能在不同位置命名是否一致 | 1-10 |

#### 功能价值传达

| 维度 | 评审要点 | 评分 |
|------|----------|------|
| 3 秒理解价值 | 用户是否能在 3 秒内理解功能价值 | 1-10 |
| 聚焦用户收益 | 功能描述是否聚焦用户收益（而非功能特性） | 1-10 |
| 视觉示例辅助 | 复杂功能是否有视觉示例辅助理解（截图/动图/交互预览） | 1-10 |
| 付费功能标注 | 付费功能是否有明确标注（标签/图标/文案提示） | 1-10 |

#### 功能可发现性

| 维度 | 评审要点 | 评分 |
|------|----------|------|
| 预期位置 | 功能是否在用户预期位置（符合平台惯例） | 1-10 |
| 引导机制 | 新功能是否有引导机制（红点/tooltip/引导气泡） | 1-10 |
| 搜索覆盖 | 功能是否可被搜索覆盖（设置/功能搜索） | 1-10 |
| 多重入口 | 高频功能是否有多重入口（快捷方式/右键菜单/搜索） | 1-10 |

### S/W/F 输出格式

```markdown
### 功能展示合理性

**Strength**
- [维度] 具体优点（引用具体功能/布局/文案作为证据）

**Weakness**
- [维度] 具体问题（引用具体功能/布局/文案作为证据）

**Fix**
- [维度] 修复建议（具体可操作的改进方案）
```

**示例**：

```markdown
### 功能展示合理性

**Strength**
- 功能优先级：核心"创建文档"按钮位于首屏视觉焦点区，使用品牌色填充 + 48px 高度，最显眼
- 功能分组：侧边栏按"最近/收藏/共享/回收站"逻辑分组，符合用户心智模型
- 功能命名：使用用户语言（"分享"而非"权限分发"，"回收站"而非"软删除列表"）

**Weakness**
- 功能优先级：设置页面单屏展示 15 个功能入口，超出 9 个上限，用户难以快速定位
- 功能命名：同一"导出"功能在列表页叫"导出"，在详情页叫"下载"，命名不一致
- 功能价值传达："智能排版"功能仅有名称无说明，用户无法 3 秒内理解其价值
- 功能可发现性："快捷键"功能隐藏在设置三级菜单，无任何引导机制，新用户难以发现
- 付费功能标注：3 个付费功能无任何标注，用户点击后才弹出付费弹窗，体验突兀

**Fix**
- 功能优先级：设置页面分组为 4 个类别（账户/通用/隐私/高级），每组 ≤4 项，单屏功能数 ≤9
- 功能命名：统一"导出"命名，全局搜索替换"下载"为"导出"（或根据语境明确区分两者含义）
- 功能价值传达："智能排版"添加副标题"自动调整段落间距和对齐方式"，3 秒内传达价值
- 功能可发现性：首次使用时在编辑器底部显示快捷键提示气泡，设置中添加可搜索的快捷键列表
- 付费功能标注：付费功能添加"PRO"标签（品牌色小标签），hover 时显示"升级解锁"提示
```

### 综合评分计算

综合评分 = Σ(各维度评分 × 权重)

**计算示例**（移动端 App）：

```
视觉审美：8 × 8% = 0.64
交互体验：7 × 15% = 1.05
品牌一致性：6 × 5% = 0.30
可访问性：5 × 10% = 0.50
商业转化：7 × 2% = 0.14
用户体验：8 × 15% = 1.20
使用流程：6 × 8% = 0.48
功能使用旅程：7 × 15% = 1.05
UI 界面设计：7 × 15% = 1.05
功能展示合理性：6 × 7% = 0.42

综合评分 = 0.64 + 1.05 + 0.30 + 0.50 + 0.14 + 1.20 + 0.48 + 1.05 + 1.05 + 0.42 = 6.83
```
