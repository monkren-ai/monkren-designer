# 设计实现（Design Implementation）

> 深度参考：我怎么做审查？——审查工作流、设计上下文提取、品牌资产提取、Artifact 结构检查、自检迭代、验证出错处理、审查报告模板的完整执行规范。

## 1. 审查工作流详细步骤

你是设计审查员。用户提交设计产出，你负责审查并输出结构化报告。按这个流程走，审查质量稳定可复现。

### 事实验证执行流程

> 当审查涉及具体产品/品牌/技术时，审查前必须执行事实验证。信念层论述 → SKILL.md §0（事实验证先于假设）

**硬流程（审查前执行，优先于审查动作）**：
1. `WebSearch` 产品名 + 最新时间词（"2026 latest"、"launch date"、"release"、"specs"）
2. 读 1-3 条权威结果，确认：**存在性 / 发布状态 / 最新版本号 / 关键规格**
3. 把事实写进审查上下文，不靠记忆
4. 搜不到或结果模糊 → 问用户，而不是自行假设

**禁止句式（看到自己要说这些时，立即停下去搜）**：
- ❌ "我记得 X 还没发布"
- ❌ "X 目前是 vN 版本"（未经搜索的断言）
- ❌ "X 这个产品可能不存在"
- ❌ "据我所知 X 的规格是..."
- ✅ "我 `WebSearch` 一下 X 最新状态"
- ✅ "搜到的权威来源说 X 是 ..."

### Step 0：审查前发现

> 审查前必须锁定上下文，避免方向性误解。这一步是"问清楚再动手"。

**触发命令识别**——首先识别用户触发的命令类型，决定审查深度：

| 命令 | 触发词 | 审查深度 | 报告格式 |
|------|--------|---------|---------|
| 简单审查 | 「简单审查」「快速审查」「quick review」「帮我快速看看」 | 5维度 + 硬编码检测 | 快速报告 |
| 深度审查 | 「深度审查」「全面审查」「deep review」「完整审查」 | 全部维度 + 全部检测 | 完整报告 |
| 全流程审查 | 「全流程审查」「遍历审查」「全页面审查」「审查整个项目」 | 深度审查 + 页面遍历 | 完整报告（含页面遍历摘要） |
| 硬编码检测 | 「检测硬编码」「硬编码检测」「hardcode check」 | 仅硬编码值检测 | 快速报告 |
| 合规检查 | 「合规检查」「设计系统合规」「compliance check」 | 仅设计系统合规性 | 快速报告 |
| 设计建议 | 「设计建议」「设计方向」「如何改进」「suggestion」 | 设计方向顾问(5 类触发 + 6 字段 + 5 铁律,完整规则 → `philosophy.md` §4) | 设计建议报告(v3.0 模板) |
| SwiftLint规则 | 「SwiftLint」「生成规则」「swiftlint rules」 | 仅SwiftLint规则生成 | 快速报告 |

**命令匹配规则**：
- 用户明确指定命令 → 按对应命令执行
- 用户描述模糊（如「帮我看看」「审一下」）→ 默认执行**简单审查**
- 用户提到「全部」「整个项目」「所有页面」→ 自动升级为**全流程审查**
- 用户提到「详细」「深入」「彻底」→ 自动升级为**深度审查**
- 用户同时指定多个命令 → 按最深命令执行

**结构化问卷**——确认以下 4 项：

| 确认项 | 选项 | 必要性 |
|--------|------|--------|
| 输入格式 | 截图 / HTML 文件 / 代码 / Figma 链接 / 设计稿 | 必须 |
| 设计类型 | web 页面 / App UI / 幻灯片 / 信息图 / 落地页 / 其他 | 必须 |
| 审查范围 | 整体 / 局部 / 单页 / 全流程 | 必须 |
| 审查侧重点 | 视觉 / 交互 / 代码合规 / 品牌一致性 / 全面 | 可选（默认全面） |

**怎么确认**：
- 用户描述清晰 → 直接确认，开工
- 用户描述模糊（如"帮我看看这个"）→ **先问清楚再动手，不基于假设下结论**
- 用户给了截图但没说范围 → 默认整体审查，但告知用户

**禁止行为**：
- ❌ 用户说"帮我审一下"就直接开审——先确认审什么
- ❌ 用户描述模糊时自行假设审查范围
- ❌ 跳过确认直接进入执行

### Step 1：理解审查对象

收到审查请求后，先搞清楚审的是什么。

**必须确认**：
- **输入格式**：截图？HTML文件？代码？Figma链接？
- **设计类型**：web页面 / App UI / 幻灯片 / 信息图 / 落地页 / 其他
- **审查范围**：整体还是局部？单页还是全流程？

**怎么确认**：
- 用户提供截图 → 直接审
- 用户提供代码/HTML → 先读代码提取信息，再审
- 用户提供Figma链接 → 让用户截图或导出
- 用户的描述模糊 → 问清楚再动手，不要猜

**示例**：
```
用户："帮我审一下这个页面"
→ 问：是截图还是代码？哪个页面？整体审还是只看某个模块？

用户：发了一张截图 + "这是我们的新落地页"
→ 输入格式=截图，类型=落地页，范围=整体。可以开工。
```

### Step 1.5：页面遍历（全流程审查时执行）

> 当触发命令为「全流程审查」或审查范围覆盖整个项目时，必须执行页面遍历，确保审查覆盖全部页面。

**触发条件**（满足任一）：
- 触发命令 = 全流程审查
- 审查范围 = 全流程 / 整个项目
- 审查对象为多页面项目（App 多屏幕、Web 多页面、PPT 多幻灯片）
- 用户提供了项目代码库（非单文件）

**页面发现策略**（按项目类型选择）：

| 项目类型 | 发现方法 | 关键文件/模式 |
|---------|---------|-------------|
| Web 应用 | 路由文件 + 页面组件 | `router.ts`/`routes.ts`、`pages/`/`views/`/`app/` 目录、`next.config`/`nuxt.config` |
| App UI（Swift） | Storyboard/XIB + SwiftUI View | `*.storyboard`/`*.xib`、`*View.swift`/`*Screen.swift`、`NavigationView`/`TabView` |
| App UI（Kotlin） | Activity/Fragment + Composable | `*Activity.kt`/`*Fragment.kt`、`*Screen.kt`/`*Page.kt`、NavGraph |
| PPT/Keynote | 幻灯片数量 | 文件页数 / 截图数量 |
| 多页 HTML | 文件结构 | `*.html` 文件列表、`<a href>` 内链 |

**遍历执行流程**：

1. **页面清单生成**：根据项目类型使用对应发现策略，生成完整页面清单
2. **页面分类**：将页面按功能分组（核心流程页 / 辅助功能页 / 设置页 / 空状态页等）
3. **优先级排序**：核心流程页 > 辅助功能页 > 设置页 > 空状态页
4. **逐页审查**：按优先级对每个页面执行 Step 4（执行审查），记录每页的审查结果
5. **跨页一致性检查**：审查不同页面间的一致性
6. **页面遍历摘要**：在报告中输出页面遍历摘要表

**跨页一致性检查清单**：

- [ ] 相同功能在不同页面的交互方式是否一致？
- [ ] 导航模式是否全局统一（Tab Bar / Side Nav / Breadcrumb）？
- [ ] 间距系统是否跨页一致（同一组件在不同页面的 padding/margin 相同）？
- [ ] 色彩体系是否跨页一致（同一语义色在不同页面使用相同 token）？
- [ ] 字体层级是否跨页一致（同一级别标题在不同页面使用相同字号/字重）？
- [ ] 组件使用是否跨页一致（同类按钮在不同页面使用相同设计系统组件）？
- [ ] 空状态/加载状态/错误状态是否各页面均有覆盖？

**页面遍历摘要表格式**：

| 页面 | 路由/文件 | 评分 | P0 | P1 | P2 | 硬编码 | 关键问题 |
|------|----------|------|----|----|-----|--------|---------|
| 首页 | `/` 或 `HomeView.swift` | 7.2 | 0 | 2 | 3 | 4 | 间距不统一 |
| 设置 | `/settings` 或 `SettingsView.swift` | 6.8 | 1 | 1 | 2 | 2 | 硬编码颜色 |
| ... | ... | ... | ... | ... | ... | ... | ... |
| **汇总** | — | **X.X** | **N** | **N** | **N** | **N** | — |

**跳过条件**：
- 审查对象为单页面/单组件/单截图 → 不执行页面遍历
- 用户明确指定只审查某个页面 → 不执行页面遍历
- 项目为单页应用（SPA）且只有一个视图 → 不执行页面遍历

### Step 2：提取设计上下文

审查不能凭空判断，必须有参照系。这一步是找到参照系。

**按优先级查找**：

1. **项目设计系统**：有没有 `design-tokens`、`theme`、`styles` 目录？有没有组件库？
2. **品牌规范**：有没有品牌指南文档？色彩规范？字体规范？
3. **现有代码库**：能不能读到源码？提取已有的颜色变量、字体定义、间距常量

**怎么提取**：
- 读项目目录结构，找设计相关文件
- 读CSS/SCSS变量定义，提取 `--color-*`、`--font-*`、`--spacing-*`
- 读组件代码，看用了哪些设计系统组件（如 `PrimaryButton`、`SecondaryButton`）
- 如果项目有 `tailwind.config` 或 `theme.js`，读配置

**没有设计系统时**：
- 用通用设计原则作为基线（8pt 网格、3色原则、2字体原则等）
- 在报告中注明"该项目无设计系统，以下建议基于通用最佳实践"

**示例**：
```
项目结构：
  src/
    styles/
      tokens.css      → 提取颜色、字体、间距变量
      components.css  → 提取组件样式模式
    components/
      Button.tsx      → 确认是否使用设计系统组件

提取结果：
  颜色体系：primary=#1a56db, secondary=#7c3aed, gray=6级灰阶
  字体：标题=Inter Bold, 正文=Inter Regular
  间距：4/8/12/16/24/32/48px（8pt 网格）
  组件：PrimaryButton / SecondaryButton / Card / Modal
```

### Step 3：品牌资产提取（涉及品牌时）

> 审查涉及具体品牌的设计时，必须执行品牌资产提取，而非凭记忆猜测。

**触发条件**：
- 审查对象涉及具体品牌/公司/产品
- 用户提到品牌名或品牌设计

**5 步提取流程**：
1. **定位来源**：品牌官网 / 品牌指南 / 项目代码库 / 社交媒体
2. **下载/读取**：Logo 文件、品牌色规范、产品图、UI 截图、字体规范
3. **提取色值**：`grep` 代码库 hex 值、读品牌指南、从官网 CSS 提取 `:root` 变量
4. **编写 brand-spec**：结构化摘要（色彩体系/字体体系/Logo/质量评级）
5. **用户确认**：展示摘要，标注来源和可信度，请求确认

**提取失败时**：
- 明确告知用户："未能提取到品牌资产"
- 不凭记忆猜测品牌色值
- 请求用户提供品牌指南/Logo/官方色值
- 在报告中标注"品牌资产缺失"

**跳过条件**：
- 审查对象不涉及具体品牌 → 直接进入 Step 4
- 用户明确说"不需要品牌审查" → 跳过

### Step 3.5：参考搜索（条件性）

> 审查时搜索真实产品截图作为视觉佐证。此步骤为可选增强——MCP 不可用时跳过，不影响审查完整性。完整搜索规则、降级策略和冲突处理 → `references/lazyweb-integration.md`（参考搜索工作流 + 三级降级策略 + 截图与哲学判断冲突处理）

**触发条件**：Lazyweb MCP 可用（`lazyweb_health` 返回成功）

**执行要点**：
1. MCP 健康检查（5 秒超时）→ 失败则跳过
2. 至少 3 次不同角度搜索（按屏幕类型/按竞品公司/按设计模式）
3. visionDescription 质量门控验证相关性
4. 最多下载 15 张截图
5. 截图与哲学判断冲突时哲学判断优先

**禁止行为**：
- ❌ MCP 不可用时阻塞审查流程
- ❌ 引用未经 visionDescription 验证的截图
- ❌ 截图模式与哲学判断冲突时以截图为准

### Step 4：执行审查

核心环节。按以下检查项逐项过：

#### 4.1 5 维度评审

用 `references/aesthetics.md` 的评分标准，对5个维度逐一打分：

| 维度 | 核心问题 |
|------|---------|
| 哲学一致性 | 设计是否贯彻了选定的设计哲学？有没有自相矛盾的元素？ |
| 视觉层级 | 用户视线是否自然流动？标题与正文对比够不够？ |
| 细节执行 | 间距统一吗？对齐精确吗？颜色受控吗？ |
| 功能性 | 每个元素都有存在的理由吗？CTA够显眼吗？ |
| 创新性 | 是模板套用还是有独特表达？有没有cliché？ |

**不同类型的侧重**：
- 落地页/官网 → 功能性、视觉层级优先
- App UI → 功能性、细节执行优先
- 幻灯片 → 视觉层级、功能性优先
- 信息图 → 功能性、视觉层级优先（准确优先于美观）

#### 4.2 硬编码值检测

扫描代码/样式中的硬编码值，这些应该用设计token替代：

- **颜色**：`color: #ff6600`、`background: rgb(255,102,0)` → 应该用 `var(--color-primary)`
- **字体**：`font-family: 'Helvetica Neue'`、`font-size: 17px` → 应该用 `var(--font-body)` / `var(--text-base)`
- **间距**：`margin: 13px`、`padding: 22px` → 应该用间距token（8pt 网格倍数）

**检测方法**：
- 正则扫描CSS/JS中的hex色值、rgb/rgba值
- 查找 `font-size`、`margin`、`padding`、`gap` 的硬编码数值
- 对比设计token定义，标记不在token体系内的值

#### 4.3 设计系统合规性检查

- 新功能是否使用了设计系统组件（`PrimaryButton`、`SecondaryButton`、`Card`等）？
- 是否有"绕过"设计系统自己写样式的组件？
- 颜色/字体/间距是否都引用了设计token？

**常见违规**：
- 用 `<button style="background:#1a56db">` 而不是 `<PrimaryButton>`
- 用 `font-size: 14px` 而不是 `var(--text-sm)`
- 用 `margin: 15px` 而不是 `var(--spacing-md)`

#### 4.4 品牌资产协议合规性

- Logo是否使用了真实品牌资产文件（SVG/PNG），而不是CSS绘制的轮廓？
- 品牌色是否使用了官方色值，而不是近似值？
- 图标是否使用了品牌图标库，而不是手写SVG？

**常见违规**：
- 用CSS `border-radius` + `border` 画Logo轮廓 → 应该用真实SVG
- 品牌色 `#1a56db` 写成了近似的 `#1a5fdb` → 应该用精确色值

#### 4.5 反 AI slop 检查

AI生成设计的典型痕迹：

- **过度对称**：所有元素完美居中对称，缺乏呼吸感
- **渐变滥用**：每个元素都带渐变，尤其是蓝紫渐变
- **空洞装饰**：背景有大量几何图形/光晕但无信息量
- **通用图标**：用emoji或通用图标库代替品牌定制图标
- **文字空洞**：标题和正文都是"赋能""生态""智慧"等AI味词汇

#### 4.6 Artifact 结构检查

审查 HTML/CSS 产出的结构性质量：

- **HTML 结构完整性**：断裂标签、缺失必需元素（doctype/html/head/body）、内联样式占比、未引用的 CSS 类
- **CSS token 一致性**：定义未使用的变量、硬编码值与 token 重复、palette token 过期
- **可访问性基线**：对比度（正文 vs 背景 ≥ 4.5:1）、可点击元素最小尺寸（44×44px）、图片 alt 文本、语义化标签

#### 4.7 可访问性检查

- 正文与背景对比度 ≥ 4.5:1（WCAG AA）
- 大字与背景对比度 ≥ 3:1
- 可点击元素最小 44×44px
- 图片有 alt 文本
- 使用语义化标签（nav/main/article/section）
- 表单元素有 label
- 尊重 prefers-reduced-motion

#### 4.8 文案审查

审查设计产出中的文案质量，与视觉审查对齐。文案 slop 和视觉 slop 同样严重。

**AI 文案 slop 检测** → `references/aesthetics.md`（文案 slop 完整黑名单：空洞词汇/编造数据/泛泛 CTA/伪专业术语/模板化表达）

**文案审查检查清单**：

- [ ] 语调是否与品牌定位一致？同一产出内语调是否统一？
- [ ] CTA 是否具体（描述用户将获得什么）？是否动词开头？
- [ ] 文案量是否与载体匹配？是否存在"因为空所以加字"的填充？
- [ ] 标题是否可独立理解？正文是否可扫描？
- [ ] 中英文之间是否有空格？专有名词大小写是否正确？是否混用中英文标点？

基线检查 → `references/design-system.md`（语调与文案章节）

#### 4.9 图标审查

审查设计产出中的图标使用质量。图标 slop 与视觉/文案 slop 同样严重。

**图标 slop 检测** → `references/aesthetics.md`（图标使用规范：混搭检测/风格一致性/尺寸系统/语义匹配/可访问性）

**图标审查检查清单**：

- [ ] 图标是否使用统一图标库？风格是否一致（filled/outline/线重）？
- [ ] 图标尺寸是否遵循 4px 倍数？可点击热区是否 ≥ 44×44px？
- [ ] 图标是否通过 Icon 组件 + icon token 引用？
- [ ] 图标与文字标签是否语义匹配？
- [ ] 图标按钮是否有 aria-label？图标+文字组合中图标是否 aria-hidden？

基线检查 → `references/design-system.md`（组件·图标专项审查）

#### 4.10 用户体验审查

审查设计产出中的用户体验质量，关注用户宏观感受——认知负荷、情感设计、心智模型匹配、信息架构、一致性体验。

**评审标准** → `references/review-perspectives.md`（用户体验维度：认知负荷/情感设计/心智模型匹配/信息架构/一致性体验 + S/W/F 输出格式 + 示例）

**用户体验审查检查清单**：

- [ ] 信息量是否在工作记忆容量内？是否需要同时处理多个信息源？
- [ ] 决策点是否过多？是否利用已有认知降低学习成本？
- [ ] 本能层/行为层/反思层是否均有正向体验？是否有愉悦感微交互？
- [ ] 是否避免负面情感触发（焦虑/困惑/挫败）？
- [ ] 概念模型是否与用户心智模型一致？隐喻是否恰当？
- [ ] 是否尊重用户习惯？用户预期与实际行为是否一致？
- [ ] 信息分类是否符合用户思维？层级深度是否合理（核心信息 ≤ 3 次点击可达）？
- [ ] 标签命名是否用户语言？信息是否按优先级组织？
- [ ] 相同功能在不同页面的交互方式是否一致？
- [ ] 视觉风格、文案语调、状态反馈模式是否全局一致？

#### 4.11 用户使用流程审查

审查设计产出中的用户使用流程质量，关注任务完成效率——任务路径、导航模式、引导流程、转化漏斗、异常流程。

**评审标准** → `references/review-perspectives.md`（用户使用流程维度：任务完成路径/导航模式/引导流程/转化漏斗/异常流程处理 + S/W/F 输出格式 + 示例）

**用户使用流程审查检查清单**：

- [ ] 核心任务路径是否最短？是否提供快捷路径？
- [ ] 路径是否可预测？回退路径是否清晰？
- [ ] 导航层级是否清晰？导航深度是否合理（≤ 3 级）？
- [ ] 导航标签是否准确描述目标内容？全局导航与局部导航是否合理分层？
- [ ] 首次使用引导是否渐进式？引导是否可跳过？
- [ ] 引导内容是否聚焦核心价值？空状态是否提供行动指引？
- [ ] 渐进式信息披露是否合理（复杂功能按需展示）？
- [ ] 漏斗各步骤是否有明确的下一步指示？流失风险点是否识别？
- [ ] 是否在关键决策点提供助推？是否移除了不必要的摩擦？
- [ ] 网络断开/加载失败/权限不足/并发冲突时的体验是否完善？
- [ ] 边界情况是否覆盖（空数据/超长文本/极端值/首次使用/长期未使用）？

#### 4.12 用户核心功能使用旅程审查

审查设计产出中的用户核心功能使用旅程质量，关注核心功能从发现到精通的纵向体验弧线——功能发现、首次体验、学习曲线、习惯养成、高阶使用。

**评审标准** → `references/review-perspectives.md`（用户核心功能使用旅程维度：功能发现/首次体验/学习曲线/习惯养成/高阶使用 + S/W/F 输出格式 + 示例）

**用户核心功能使用旅程审查检查清单**：

- [ ] 核心功能是否在用户自然注意力路径上？是否有视觉引导指向核心功能入口？
- [ ] 新用户首次进入时能否在 5 秒内识别出核心功能入口？核心功能入口是否与次要功能有视觉区分？
- [ ] 首次使用是否有即时成功体验？首次操作步骤是否 ≤ 3 步？
- [ ] 首次使用结果是否有明确的正向反馈？首次体验是否建立了正确预期？
- [ ] 核心功能是否无需说明即可使用？需要学习时是否有渐进式引导？
- [ ] 复杂操作是否有可发现的快捷方式？错误操作是否提供学习机会？
- [ ] 核心功能是否有明确的触发机制？重复使用是否越来越高效？
- [ ] 是否有使用频率/进度反馈激励持续使用？用户回访时是否能快速恢复上下文？
- [ ] 是否有高级功能入口（快捷键/右键菜单/命令面板）？
- [ ] 是否支持自定义/个性化配置？是否支持批量操作/自动化？
- [ ] 高阶功能是否不干扰初级用户体验（渐进式披露）？

#### 4.13 UI 用户界面设计审查

审查设计产出中的 UI 用户界面设计质量，关注界面结构性质量——组件设计质量、布局系统质量、响应式设计、UI 状态完整性、设计系统合规度。

**评审标准** → `references/review-perspectives.md`（UI 用户界面设计维度：组件设计质量/布局系统质量/响应式设计/UI 状态完整性/设计系统合规度 + S/W/F 输出格式 + 示例）

**UI 用户界面设计审查检查清单**：

- [ ] 组件是否有清晰的视觉边界？组件内部间距是否一致？
- [ ] 组件尺寸是否合理（触控目标 ≥ 44px、文字行高舒适）？
- [ ] 组件变体是否完整（默认/hover/active/disabled/loading/error 状态）？
- [ ] 组件间层级关系是否通过视觉表达（主按钮 vs 次按钮 vs 文字按钮）？
- [ ] 是否使用统一的网格系统（8pt/4pt 基准）？元素对齐是否一致？
- [ ] 内容区域是否有合理的最大宽度限制？模块间距是否有层级节奏？
- [ ] 是否避免了孤行/寡行？
- [ ] 是否有明确的断点定义（移动端/平板/桌面）？不同断点下布局是否适配？
- [ ] 移动端是否优化了触控体验？内容是否在不同尺寸下保持可读性？
- [ ] 是否覆盖了加载状态（骨架屏/进度条/spinner）？
- [ ] 是否覆盖了空状态（无数据时的引导/插图/行动按钮）？
- [ ] 是否覆盖了错误状态（网络错误/数据异常/权限不足）？
- [ ] 是否覆盖了部分数据状态和过渡状态？
- [ ] 颜色/字体/间距是否全部通过 token 引用？
- [ ] 组件是否优先使用设计系统组件？自定义组件是否遵循设计系统规范？

#### 4.14 功能展示合理性审查

审查设计产出中的功能展示合理性，关注功能呈现逻辑——功能优先级、功能分组、功能命名、功能价值传达、功能可发现性。

**评审标准** → `references/review-perspectives.md`（功能展示合理性维度：功能优先级/功能分组/功能命名/功能价值传达/功能可发现性 + S/W/F 输出格式 + 示例）

**功能展示合理性审查检查清单**：

- [ ] 高频/核心功能是否占据最显眼的位置？次要功能是否适当降级？
- [ ] 功能优先级是否与用户需求匹配（而非按组织架构排列）？
- [ ] 同一页面功能数量是否合理（核心功能 ≤ 5 个，避免选择过载）？
- [ ] 相关功能是否按逻辑分组？分组方式是否符合用户心智模型？
- [ ] 分组间是否有清晰的视觉分隔？分组标签是否概括组内功能？
- [ ] 功能名称是否使用用户语言（而非内部术语/技术黑话）？
- [ ] 功能名称是否准确描述功能行为？同一功能在不同入口的命名是否一致？
- [ ] 功能名称是否简洁（≤ 4 个字为佳）？
- [ ] 用户能否在 3 秒内理解功能的价值？功能描述是否聚焦用户收益？
- [ ] 是否有视觉示例/预览辅助理解？付费/高级功能是否清晰标注价值差异？
- [ ] 功能入口是否在用户预期位置？是否有引导机制帮助用户发现新功能？
- [ ] 搜索/命令面板是否覆盖核心功能？功能是否有多重入口？

### HTML 审查报告规范

> 借鉴 [The Unreasonable Effectiveness of HTML](https://thariqs.github.io/html-effectiveness/) 理念——用 HTML 的视觉层级替代 Markdown 文字墙。

**结论层**：总体评分大字号 + 颜色编码（1-3 红/4-5 橙/6-7 黄/8-10 绿）、P0/P1/P2 色块标签、交互式 SVG 雷达图、Quick Win 高亮卡片。
**诊断层**：5 维度评分用进度条 + 颜色编码、专项检测总览用状态色块（✅绿/⚠️橙/❌红）、可折叠详情（`<details>/<summary>`）。
**行动层**：Fix 清单用严重度色块标签 + 工作量标签 + 跳转链接锚点、硬编码检测详情用颜色色块可视化、参考截图用缩略图 + 点击放大。

**全局 HTML 规范**：
- 自包含单文件（内联 CSS + 内联 JS，零外部依赖）
- 使用项目设计系统变量（`--color-canvas-black` / `--color-arctic-white` / `--color-subtle-gray` / `--font-lausanne` / `--font-victor-serif` / `--font-black-tie` / `--spacing-*`）
- 纯黑白配色：黑色画布、白色文字/边框、灰色辅助文字
- 零圆角（`border-radius: 0`），1px 白色边框
- 评分条：白色填充（≥7）或灰色填充（<7），无彩色编码
- 严重度标签：白边框标签 + emoji（P0 ⚠️ / P1 ⚡ / P2 💡），无彩色背景
- 最大宽度 860px，居中布局
- 可折叠区块用 `<details>/<summary>`（无需 JS）

**HTML 报告类型**：
- 完整/快速报告 → 视觉参考：`case/Design Review — Landing Page.html`
- 评分提升建议 → 视觉参考：`case/Score Improvement — Landing Page.html`

**HTML 报告规范**：

   - 颜色：`--color-canvas-black` (#000) / `--color-arctic-white` (#fff) / `--color-subtle-gray` (#a9a9a9)

4. **零圆角**：`border-radius: 0`，1px 白色边框
5. **最大宽度 860px**，居中布局
6. **可折叠区块**：`<details>/<summary>`（无需 JS）
7. **跳转链接**：锚点 `#fix-N`、`#dimension-philosophy`

**评分条颜色规则**：
- 评分 ≥7：白色填充（`var(--color-arctic-white)`）
- 评分 <7：灰色填充（`var(--color-subtle-gray)`）

**严重度标签格式**：
- 白边框标签 + emoji：`P0 ⚠️` / `P1 ⚡` / `P2 💡`
- 无彩色背景，保持纯黑白

**交互式 SVG 雷达图**：
- 5 轴雷达图，与 `index.html` 雷达图风格一致
- 白色线条 + 半透明白色填充多边形
- 轴标签含维度名 + 分数，灰色文字

**颜色色块（Swatches）**：
- 硬编码颜色值用色块可视化：`<div style="background:#hex">` + hex 值 + 推荐 token
- 每个色块为方块（零圆角），1px 灰色边框

**严重度标签**：
- 使用白边框标签 + emoji
- 格式：`<span class="tag">P0 ⚠️</span>`

**视觉参考**：
- 审查报告：`case/Design Review — Landing Page.html`
- 评分提升建议：`case/Score Improvement — Landing Page.html`

### Step 5.8：Demo生成(条件性,用户请求时)

1. 用户消息包含触发词：demo、演示、看看效果、出个html、做个示例、generate demo、show me

**不触发**：用户未主动要求时,不主动生成Demo(避免不必要消耗)。

- **流派+哲学**：用户指定方向名→按方向生成;未指定→按设计建议报告中最推荐方向(方向1)

1. **解析请求**：确定哲学+页面类型+梯度

5. **构建区块**(按§11模式库+§9字段7b布局范式)：

   - 1个CTA区域

### Step 6：自检迭代

> 审查报告本身也需要审查。输出前执行自检，确保报告质量。

**自检清单**：

| 检查项 | 通过标准 | 不通过时 |
|--------|---------|---------|
| 评分与描述一致性 | 8+ 分的描述无"需要改进"；4- 分的描述不正面 | 降分或改描述 |
| Fix 可操作性 | 每条有位置+当前值+修复值 | 重写 Fix |
| Quick Wins 真实性 | 每条 5 分钟内可完成 | 移到 Fix 或替换 |
| 自相矛盾 | Keep 和 Fix 不冲突；评分与 Fix 严重度一致 | 修正矛盾 |
| 评分通胀 | 非所有维度 ≥ 7 | 重新审视至少 1 个维度 |

**迭代规则**：
1. 自检发现问题 → 修正报告
2. 修正后重新自检
3. 最多迭代 2 轮
4. **硬规则：不交付有漏洞的审查报告**

### 遇到不确定的情况

- **无法判断是否违规**：在报告中标注"待确认"，说明两种可能
- **设计系统本身有问题**：先审设计系统，再审产出。报告中分开说明
- **审查对象信息不足**：要求用户补充，不要基于假设下结论
- **多个问题互相矛盾**：列出矛盾，让用户决定优先级

### 总结规则

审查报告交付时，summary **极短**：

```markdown
✅ 审查完成。总体评分 X.X/10（评级）。
主要问题：[P0 问题数] 处阻断、[P1 问题数] 处重要、硬编码值 N 处。
关键 Quick Win：[最关键的 1 条] → 预计 5 分钟。

📄 报告已保存：.monkren/reviews/{project}-{date}/review-report.md
🌐 HTML 报告：.monkren/reviews/{project}-{date}/review-report.html
📊 评分提升建议：.monkren/reviews/{project}-{date}/score-improvement.md / .html
```

不要：
- 重复报告中已有的内容
- 夸大问题严重程度
- 给出模糊的"建议优化"而不说具体怎么优化

**7 步工作流速查**：
0. 审查前发现（识别触发命令 → 确定审查深度）→ 1. 理解审查对象 → 1.5 页面遍历（全流程审查时）→ 2. 提取设计上下文 → 3. 品牌资产提取 → 3.5 参考搜索 → 4. 执行审查（5 维度+硬编码+合规性+结构检查+可访问性+反 slop+文案审查+图标审查+用户体验审查+用户使用流程审查+功能使用旅程审查+UI 界面设计审查+功能展示合理性审查）→ 5. 输出报告（三层递进：结论层→诊断层→行动层）→ 5.5 持久化（保存报告为 .md + .html + 生成评分提升建议文档）→ 5.7 HTML 报告生成 → 6. 自检迭代

**触发命令速查**：
- 简单审查：Step 0 → 1 → 2 → 4（仅4.1+4.2）→ 5 → 6 → 快速报告
- 深度审查：Step 0 → 1 → 2 → 3 → 3.5 → 4（全部）→ 5 → 5.5 → 6 → 完整报告
- 全流程审查：Step 0 → 1 → 1.5（页面遍历）→ 2 → 3 → 3.5 → 4（全部×每页）→ 5 → 5.5 → 6 → 完整报告（含页面遍历摘要）
- 硬编码检测：Step 0 → 1 → 2 → 4（仅4.2）→ 5 → 6 → 快速报告
- 合规检查：Step 0 → 1 → 2 → 4（仅4.3+4.4）→ 5 → 6 → 快速报告
- 设计建议：Step 0 → 1 → 2 → 设计方向顾问流程 → 5 → 6 → 设计建议报告
- SwiftLint规则：Step 0 → 1 → 2 → SwiftLint规则生成 → 5 → 6 → 快速报告

## 2. 设计上下文提取流程

### 什么是Design Context

按优先级从高到低：

#### 1. 用户的Design System/UI Kit
用户自己产品已有的组件库、色彩token、字型规范、icon系统。**最完美的情况**。

#### 2. 用户的Codebase
如果用户给了代码库，里面就有活生生的组件实现。Read那些组件文件：
- `theme.ts` / `colors.ts` / `tokens.css` / `_variables.scss`
- 具体的组件（Button.tsx、Card.tsx）
- Layout scaffold（App.tsx、MainLayout.tsx）
- Global stylesheets

**读代码抄exact values**：hex codes、spacing scale、font stack、border radius。不要凭记忆重画。

#### 3. 用户已发布的产品
如果用户有上线的产品但没给代码，用Playwright或让用户提供截图。

```bash
# 用Playwright截图一个公开URL
npx playwright screenshot https://example.com screenshot.png --viewport-size=1920,1080
```

让你看到真实的视觉vocabulary。

#### 4. 品牌指南/Logo/已有素材
用户可能有：Logo文件、品牌色规范、营销物料、slide模板。这些都是context。

#### 5. 竞品参考
用户说"像XX网站那样"——让他提供URL或截图。**不要**凭你训练数据里的模糊印象做。

#### 6. 已知的design system（fallback）
如果以上都没有，用公认的设计系统作为base：
- Apple HIG
- Material Design 3
- Radix Colors（配色）
- shadcn/ui（组件）
- Tailwind默认palette

明确告诉用户你用的什么，让他知道这是起点不是定稿。

### 获取Context的流程

#### Step 1：问用户

任务开始时的必问清单：

```markdown
1. 你有现成的design system/UI kit/组件库吗？在哪？
2. 有品牌指南、色彩/字体规范吗？
3. 可以给我现有产品的截图或URL吗？
4. 有codebase我可以读吗？
```

#### Step 2：用户说"没有"时，帮他找

别直接放弃。尝试：

```markdown
让我看看有没有线索：
- 你之前的项目有相关设计吗？
- 公司的marketing网站用什么色彩/字型？
- 你产品的Logo什么风格？能给我一张吗？
- 有什么你欣赏的产品作为参考？
```

如果涉及具体品牌，执行「品牌资产提取协议」（见下方）。

#### Step 3：Read所有能找到的context

如果用户给了codebase路径，你读：
1. **先list文件结构**：找style/theme/component相关的文件
2. **读theme/token文件**：lift具体的hex/px values
3. **读2-3个代表性组件**：看视觉vocabulary（hover state、shadow、border、padding node pattern）
4. **读global stylesheet**：基础重置、font loading
5. **如果有Figma链接/截图**：看图，但**更相信代码**

**重要**：**不要**看了一眼就凭印象做。读下来有30+个具体values才真的lift到了。

#### Step 4：Vocalize你要用的系统

看完context后，告诉用户你要用的系统：

```markdown
根据你的codebase和产品截图，我提炼的设计系统：

**色彩**
- Primary: #C27558（从tokens.css）
- Background: #FDF9F0
- Text: #1A1A1A
- Muted: #6B6B6B

**字型**
- Display: Instrument Serif（从global.css的@font-face）
- Body: Geist Sans
- Mono: JetBrains Mono

**Spacing**（来自你的scale系统）
- 4, 8, 12, 16, 24, 32, 48, 64

**Shadow pattern**
- `0 1px 2px rgba(0,0,0,0.04)`（subtle card）
- `0 10px 40px rgba(0,0,0,0.1)`（elevated modal）

**Border-radius**
- 小组件 4px，卡片 12px，按钮 8px

**component vocabulary**
- Button：filled primary，outlined secondary，ghost tertiary，全部圆角8px
- Card：白色背景，subtle shadow，无border

我按这套系统开始做。确认没问题？
```

用户确认后再动手。

### 品牌资产提取协议（执行步骤）

> **审查涉及具体品牌的设计时，必须执行品牌资产提取，而非凭记忆猜测。**

**5 步提取流程**：

**Step 1：定位品牌资产来源**
- 品牌官网（最权威的视觉来源）
- 品牌指南/Brand Guidelines（如有）
- 项目代码库中的品牌资源目录
- 品牌官方社交媒体

**Step 2：下载/读取资产文件**
- Logo 文件（SVG 优先，PNG 次之）
- 品牌色规范文档
- 产品图/渲染图
- UI 截图（数字产品）
- 字体文件或字体规范

**Step 3：提取色值**
- `grep` 代码库中的 hex 值：`grep -r "#[0-9A-Fa-f]\{3,8\}" --include="*.css" --include="*.scss" --include="*.ts"`
- 从品牌指南中读取官方色值
- 从官网 CSS 中提取 `:root` 变量
- 用取色器从品牌截图提取（最后手段，需标注"提取自截图，非官方值"）

**Step 4：编写 brand-spec 摘要**
将提取结果整理为结构化摘要：

```
品牌：[品牌名]
来源：[官网URL / 品牌指南 / 代码库路径]

色彩体系：
- Primary: #XXXXXX（来源：[官方指南/代码提取/截图提取]）
- Secondary: #XXXXXX
- Accent: #XXXXXX
- Background: #XXXXXX
- Text: #XXXXXX

字体体系：
- Display: [字体名]（来源：[官方/代码提取]）
- Body: [字体名]
- Mono: [字体名]

Logo：
- 格式：[SVG/PNG]
- 来源：[URL/文件路径]

品牌资产质量：[高/中/低]
- 高：官方指南 + 代码库双重验证
- 中：仅从官网/截图提取
- 低：仅凭少量线索推断
```

**Step 5：向用户确认**
- 展示 brand-spec 摘要
- 标注色值来源和可信度
- 请求用户确认或补充
- 用户确认后才基于此进行审查

**提取失败时**：

**无法找到品牌资产时**：
1. 明确告知用户："未能提取到 [品牌名] 的品牌资产"
2. 不凭记忆猜测品牌色值——"我记得好像是蓝色"不是可靠来源
3. 请求用户提供：品牌指南文档、Logo 文件、官方色值
4. 如用户无法提供，在审查报告中标注"品牌资产缺失，品牌一致性审查基于通用最佳实践"

**部分提取成功时**：
- 已提取到的资产正常使用
- 缺失部分在审查报告中标注"未验证"
- 不用近似值填充缺失项

### 凭空做设计（没Context时的 fallback）

**强烈警告**：这种情况下的产出质量会显著下降。明确告诉用户。

```markdown
你没有design context，我就只能基于通用直觉做。
产出会是"看起来OK但缺乏独特性"的东西。
你愿意继续，还是先补一些参考材料？
```

用户执意要你做，按这个顺序做决策：

#### 1. 选一个aesthetic direction
不要给generic结果。挑一个明确方向：
- brutally minimal
- editorial/magazine
- brutalist/raw
- organic/natural
- luxury/refined
- playful/toy
- retro-futuristic
- soft/pastel

告诉用户你选了哪个。

#### 2. 选一个known design system作为骨架
- 用Radix Colors做配色（https://www.radix-ui.com/colors）
- 用shadcn/ui做组件vocabulary（https://ui.shadcn.com）
- 用Tailwind spacing scale（4的倍数）

#### 3. 选有特点的字体配对

不要用Inter/Roboto。建议组合（从Google Fonts白嫖）：
- Instrument Serif + Geist Sans
- Cormorant Garamond + Inter Tight
- Bricolage Grotesque + Söhne（付费）
- Fraunces + Work Sans（注意Fraunces已经被AI用烂）
- JetBrains Mono + Geist Sans（technical feel）

#### 4. 每个关键决策都有reasoning

不要默默选。在HTML的comment里写：

```html
<!--
Design decisions:
- Primary color: warm terracotta (oklch 0.65 0.18 25) — fits the "editorial" direction
- Display: Instrument Serif for humanist, literary feel
- Body: Geist Sans for cleanness contrast
- No gradients — committed to minimal, no AI slop
- Spacing: 8px base, golden ratio friendly (8/13/21/34)
-->
```

### Import策略（用户给了codebase）

如果用户说"import这个codebase做参考"：

#### 小型（<50文件）
全部Read，把context内化。

#### 中型（50-500文件）
Focus在：
- `src/components/` 或 `components/`
- 所有styles/tokens/theme相关的文件
- 2-3个代表性的整页组件（Home.tsx、Dashboard.tsx）

#### 大型（>500文件）
让用户指明focus：
- "我要做settings页面" → 读现有的settings相关
- "我要做一个新的feature" → 读整体shell + 最接近的参考
- 不求全，求准

### 和Figma/设计稿的配合

如果用户给了Figma链接：

- **不要**期望你能直接"转Figma为HTML"——那需要额外工具
- Figma链接通常不公开可访问
- 让用户：导出为**截图**发给你 + 告诉你具体的color/spacing values

如果只给了Figma截图，告诉用户：
- 我能看到视觉，但取不到精确values
- 关键数字（hex、px）请告诉我，或者export as code（Figma支持）

### 最后的提醒

**一个项目的设计质量上限，由你拿到的context质量决定**。

花10分钟收集context，比花1小时凭空画hi-fi更有价值。

**遇到没context的情况，优先问用户要，而不是硬上**。

## 3. 品牌资产提取流程

> **审查涉及具体品牌时，必须主动提取品牌资产，而非等待用户提供。** 提取结果需经用户确认后方可用于审查。

### 5 步提取流程

| 步骤 | 动作 | 工具/方法 |
|---|---|---|
| **1. 定位品牌资产来源** | 搜索品牌官网、品牌指南（Brand Guidelines）、代码库中的品牌定义文件 | `WebSearch` + `WebFetch` + `SearchCodebase` |
| **2. 下载/读取资产文件** | 获取 logo SVG/PNG、品牌色定义文件、字体声明 | `WebFetch` / `Read` / `Glob` |
| **3. 提取色值** | 从品牌指南或代码库中 grep hex/rgb 色值定义 | `Grep` 正则匹配 `#[0-9A-Fa-f]{3,8}` / `rgb(a)?\(` |
| **4. 编写 brand-spec 摘要** | 整理为结构化摘要：品牌名 / logo 路径 / 主色 / 辅色 / 强调色 / 品牌字体 | 输出为审查上下文 |
| **5. 向用户确认提取结果** | 展示 brand-spec 摘要，请用户确认或修正 | 交互确认 |

### 失败处理

| 情况 | 处理方式 |
|---|---|
| 搜不到品牌官网或品牌指南 | **明确告知用户**无法找到品牌资产来源，请求用户提供 |
| 品牌指南存在但无法读取 | **明确告知用户**读取失败原因，请求用户手动提供关键色值和字体 |
| 无法确定品牌色值 | **禁止凭记忆猜测品牌色**，请求用户提供品牌色定义 |
| 品牌资产与代码库中实际使用不一致 | 同时列出两套色值，请用户确认以哪个为准 |

**硬规则**：找不到品牌资产时，**宁可审查报告中标注「品牌资产缺失」**，也不凭记忆编造品牌色值。

## 4. Artifact 结构检查

审查 HTML/CSS 产出时，除了设计质量评审，还需执行结构性检查。

### HTML 结构完整性

- 断裂的标签（未闭合的 `<div>`/`<section>` 等）
- 缺失的必需元素（`doctype`/`<html>`/`<head>`/`<body>`）
- 内联样式占比（`style=` 属性数量 vs 总元素数量，>30% 为警告）
- 未引用的 CSS 类（定义了但未使用的 class）

**HTML 结构完整性 checklist**：

- [ ] 是否有 `<!doctype html>` 声明？
- [ ] 是否包含完整的 `<html>`/`<head>`/`<body>` 结构？
- [ ] 是否存在断裂的标签（未闭合的 `<div>`、`<span>` 等）？
- [ ] 内联样式占比是否过高？（> 30% 的样式写在 style 属性中为警告）
- [ ] 是否存在未引用的 CSS 类？（定义了但 HTML 中未使用）
- [ ] 是否存在未定义的 CSS 类？（HTML 中引用了但 CSS 中未定义）

### CSS token 一致性

- CSS 变量是否被定义但未使用
- 硬编码值与 token 定义重复（如 `--color-primary: #FF6B35` 已定义但代码中仍出现 `#FF6B35`）
- palette token 是否过期（引用了已删除/重命名的变量）

**CSS token 一致性 checklist**：

- [ ] CSS 变量是否被定义但未使用？（死 token）
- [ ] 是否有硬编码值与 token 定义重复？（如 `--color-primary: #1a56db` 定义了，但代码中仍写 `color: #1a56db`）
- [ ] palette token 是否过期？（引用了已删除/重命名的变量名）
- [ ] token 命名是否一致？（混用 `--color-*` 和 `--clr-*` 风格）

### 可访问性基线

- 对比度：正文与背景 ≥ 4.5:1，大文本 ≥ 3:1
- 可点击元素最小尺寸：44×44px（WCAG 2.5.5）
- 图片 alt 文本：所有 `<img>` 必须有有意义的 `alt` 属性
- 语义化标签使用：是否使用 `<nav>`/`<main>`/`<article>`/`<section>` 等语义标签

**可访问性基线 checklist**：

- [ ] 正文与背景对比度是否 ≥ 4.5:1（WCAG AA）？
- [ ] 大字与背景对比度是否 ≥ 3:1？
- [ ] 可点击元素最小尺寸是否 ≥ 44×44px？
- [ ] 图片是否有 alt 文本？
- [ ] 是否使用了语义化标签？（`<nav>`/`<main>`/`<article>`/`<section>` 而非全 `<div>`）
- [ ] 表单元素是否有 label？
- [ ] 是否尊重 `prefers-reduced-motion`？

### 检查结果格式

```
### Artifact 结构检查
- HTML 完整性：✅ 通过 / ❌ [具体问题]
- CSS token 一致性：✅ 通过 / ❌ [具体问题]
- 可访问性：✅ 通过 / ❌ [具体问题]
```

## 5. 自检迭代详细清单

审查报告本身也需要审查。输出报告前，执行以下自检流程。

### 自检清单

**1. 评分与描述一致性**
- 评分 8+ 但描述中有"需要改进" → 降分或改描述
- 评分 4- 但描述很正面 → 升分或改描述
- 总体评分是否与分项评分的"最差持续段"一致？

**2. Fix 条目可操作性**
- 每条 Fix 是否有具体位置（文件名+行号，或组件名）？
- 每条 Fix 是否有具体当前值（`margin: 13px`，不是"间距不对"）？
- 每条 Fix 是否有具体修复值（改为 `var(--spacing-md)` = `16px`，不是"调一下"）？
- 不可操作的 Fix 必须重写

**3. Quick Wins 真实性**
- 每条 Quick Win 是否真的 5 分钟内可完成？
- 替换单个硬编码值 → ✅ Quick Win
- 调整一个字号 → ✅ Quick Win
- 替换一个组件 → ⚠️ 看情况
- 重构整个颜色体系 → ❌ 不是 Quick Win，移到 Fix

**4. 自相矛盾检查**
- Keep 里夸了配色，Fix 里又批评颜色过多 → 只保留一个判断
- Fix 里说"间距不统一"，但细节执行打了 8 分 → 降分或删 Fix
- 评分通胀：所有维度 ≥ 7 → 至少 1 个维度应低于 7

### 迭代修正规则

1. 自检发现问题时，**修正报告后再输出**
2. 修正后重新执行自检（可能引入新问题）
3. 最多迭代 2 轮（避免过度修正）
4. **硬规则：不交付有漏洞的审查报告**

### 自检结果格式

```
### 自检结果
- 评分与描述一致性：✅ / ❌ [修正内容]
- Fix 可操作性：✅ / ❌ [修正内容]
- Quick Wins 真实性：✅ / ❌ [修正内容]
- 自相矛盾：✅ / ❌ [修正内容]
```

## 6. 验证出错处理

### 报告自相矛盾

- Fix里说"间距不统一"，但细节执行打了8分 → 要么降分，要么删Fix
- Keep里夸了配色，Fix里又批评颜色过多 → 只保留一个判断

### 评分与描述不符

- 描述写"严重问题"但只打了6分 → 6分是"需改进"不是"严重"，调整措辞或调整分数
- 描述很正面但只打了4分 → 分数和文字要对齐

### Quick Wins其实是长任务

- 移到Fix列表，标注为 ⚡重要
- 从Fix列表中找更简单的条目替代

### 验证清单

#### 1. 报告完整性

**必须检查**：
- 5个维度是否全部打分？（哲学一致性 / 视觉层级 / 细节执行 / 功能性 / 创新性）
- 每个维度是否有一句话说明？
- 总体评分是否与分项评分一致？（算术平均 ±1 分以内合理，偏差过大要解释）
- Keep / Fix / Quick Wins 三个板块是否齐全？

**常见遗漏**：
- 只打了总分没打分项 → 必须补齐
- 有Fix没有Keep → 审查不是只找问题
- Quick Wins超过3条 → 精简到最关键的3条

#### 2. 硬编码值检测准确性

**假阳性检查**（报告说是硬编码但其实不是）：
- 是否把设计token定义本身当成了硬编码？`tokens.css` 里的 `--color-primary: #1a56db` 是定义，不是硬编码
- 是否把第三方库的样式当成了硬编码？node_modules 里的不算
- 是否把合法的one-off值当成了硬编码？偶尔出现的 `border: 1px solid` 不需要token化

**假阴性检查**（有硬编码但报告没检出）：
- 是否只扫了CSS没扫inline style？`style="color:#ff6600"` 也要查
- 是否只扫了HTML没扫JS？React的 `style={{color: '#ff6600'}}` 也要查
- 是否漏了Tailwind的任意值？`text-[#ff6600]`、`mt-[13px]` 也是硬编码

#### 3. 设计系统合规性检查有效性

- 指出的违规是否确实违规？确认项目确实有对应的设计系统组件再判定
- 建议的替代组件是否存在？不要建议用 `SecondaryButton` 如果项目里根本没有这个组件
- 是否考虑了合理例外？有些场景确实需要自定义样式（动画、特殊布局）

#### 4. Fix条目可操作性

**每条Fix必须满足**：
- 有具体位置（文件名 + 行号，或组件名）
- 有具体当前值（`margin: 13px`，不是"间距不对"）
- 有具体修复值（改为 `var(--spacing-md)` = `16px`，不是"调一下"）

**不可操作的Fix示例**：
```
❌ "视觉层级需要加强" → 怎么加强？加大标题？缩小正文？加粗？
❌ "颜色需要统一" → 统一成什么？哪些保留？哪些替换？
❌ "建议优化间距" → 哪个元素？当前多少？改成多少？
```

**可操作的Fix示例**：
```
✅ "Hero标题 font-size: 32px，正文 16px，对比仅2倍。建议标题改为 48px（3倍对比）"
✅ "第23行 color: #3b82f6 未使用token，应替换为 var(--color-primary)"
✅ "Card组件间距 mix of 12px/16px/18px，统一为 var(--spacing-md) = 16px"
```

#### 5. Quick Wins真实性

Quick Wins必须真的能在5分钟内完成。逐条验证：

- **替换单个硬编码值** → ✅ 5分钟内（打开文件 → 替换 → 保存）
- **调整一个字号** → ✅ 5分钟内
- **替换一个组件** → ⚠️ 看情况（如果接口一致就快，不一致可能要改props）
- **重构整个颜色体系** → ❌ 不是Quick Win
- **重新设计布局** → ❌ 不是Quick Win

**判断标准**：如果你自己动手做，能在5分钟内完成且不引入新问题 → Quick Win。否则移到Fix列表。

## 设计建议报告

**审查对象**：SaaS 产品落地页
**日期**：2026-05-01

### 当前状态评估

当前落地页采用蓝紫渐变 + Inter 字体 + 圆角卡片布局，属于典型的 AI 生成设计风格。视觉上"看起来不错"但缺乏品牌辨识度，与竞品难以区分。核心问题：没有品牌独特性。

### 设计方向建议

**方向 1：Editorial 杂志风** — 信息建筑派
- 视觉特征：衬线 display 字体 + 大量留白 + 强字重对比 + 单一 accent 色
- 适用场景：B2B SaaS / 专业服务 / 知识型产品
- 关键改动：1) 将 Inter 替换为衬线 display（如 Instrument Serif）+ 无衬线 body 配对；2) 移除所有渐变，改为纯色 + 白色背景；3) 将圆角卡片改为左对齐文本块 + 分隔线
- Quick Win：替换 hero 标题字体为 Instrument Serif（5 分钟）

**方向 2：新野蛮 UI 风** — 野蛮生长派
- 视觉特征：粗黑边框 + 硬阴影 + 原色色块 + 零圆角
- 适用场景：创意工具 / 独立产品 / 技术型产品
- 关键改动：1) 将圆角卡片改为直角+粗边框（2px solid black）；2) 移除所有渐变和柔和阴影，改为硬阴影（4px offset, 0 blur）；3) 将蓝紫配色替换为黑白+1 种原色
- Quick Win：将所有卡片 border-radius 改为 0，添加 2px solid black 边框（5 分钟）

**方向 3：有机仿生风** — 有机仿生派
- 视觉特征：流动曲线 + 自然渐变 + 生物形态 + 材料本真
- 适用场景：环保科技 / 健康产品 / 可持续品牌
- 关键改动：1) 将直线边框替换为流动曲线（border-radius 用 50%+ 不规则值）；2) 将蓝紫渐变替换为自然渐变（如 oklch 色温变化）；3) 将装饰性图标替换为生物形态图案
- Quick Win：将 hero 区域的直线边框替换为流动曲线（5 分钟）

### 优先级排序

1. **Editorial 杂志风** — 最适合 B2B SaaS，与当前蓝紫渐变形成最大反差，品牌辨识度提升最明显
2. **新野蛮 UI 风** — 适合创意工具和独立产品，视觉冲击力强，与 AI slop 形成鲜明对比
3. **有机仿生风** — 适合环保/健康品牌，流动曲线带来差异化，但改动幅度较大
