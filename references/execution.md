## 1. 报告类型与格式

| 格式 | 适用场景 | 包含内容 | 输出文件 |
|------|---------|---------|---------|
| 完整报告 | 全面审查 / 复杂设计(落地页/App UI/品牌设计) | 结论层 + 诊断层 + 行动层 | `.md` + `.html` |
| 快速报告 | 快速审查 / 简单设计(单个组件/图标/局部修改) | 结论层 + 行动层(无诊断层维度卡片和专项检测详情) | `.md` + `.html` |
| 设计建议报告 | 设计方向建议(非审查) | 当前状态评估 + 3 个方向建议(8 字段)+ 优先级 | `.md` + `.html` |
| 评分提升建议 | 审查完成后额外生成 | 各维度提升路径 + 路线图 | `.md` + `.html` |

**自动选择规则**:
- 审查范围=整体 → 完整报告
- 审查范围=局部/单页 → 快速报告
- 用户请求"设计建议""设计方向""如何改进" → 设计建议报告
- 用户明确指定格式 → 以用户选择为准

**双格式规则**:
- `.md` 用于机器可读和版本对比(git diff、搜索索引)
- `.html` 用于人类阅读和分享(浏览器打开、邮件分享)
- HTML 报告在浏览器中打开(`open review-report.html`)

## 2. 三层递进报告结构(完整报告)

完整报告采用三层递进结构,从结论到诊断到行动,让不同深度的读者都能找到所需信息。

### 2.1 结论层(1 分钟看懂)

- 总体评分(X.X/10 + 评级)
- 1 句话结论
- P0 问题数 + 硬编码值数
- 关键 Quick Win(最关键的 1 条修复 + 预计 5 分钟)
- 内联 SVG 雷达图(见 §4)

**全流程审查时附加**:
- 页面遍历摘要表(页面 / 路由 / 评分 / P0/P1/P2 / 硬编码 / 关键问题)
- 跨页一致性检查表(交互/导航/间距/色彩/字体/组件/状态覆盖 7 项)

## 3. 设计建议报告 8 字段模板

> v3.1(2026-06-26):从 v3.0 的 6 字段升级为 8 字段结构。字段 1-6 保留原有逻辑并加强引用 12 维度画像;字段 7-8 为新增可落地能力(落地概念包 + Demo 预览)。

**触发**:`设计建议` / `设计方向` / `如何改进` / `什么风格` / `suggestion` → 自动走设计方向顾问流程。

**触发阶段**(5 类场景,必须先识别):

| 触发场景 | 关键信号 | 输出重点 |
|---------|---------|---------|
| **冷启动 (0→1)** | "新项目","没 design system","从零开始" | 入门成本、组件库适配、第一个可落地方向 |
| **增量设计 (1→N)** | "已有 system,想新增模块/页面" | 风格延展性、token 复用、组件一致性 |
| **方向诊断** | "已有设计,想识别它属于什么流派" | 归类、特征对比、改进方向 |
| **瓶颈突破** | "陷同质化","想换风格","被说太普通" | 差异最大化、风险标注、迁移路径 |
| **标杆对齐** | "想对标 XX 产品/品牌" | 路径、改造幅度、可行性、所需资源 |

**Skip 条件**:
- 用户给了明确的审查标准或品牌规范 → 走审查工作流
- 用户只要代码审查 / 硬编码检测 → 走检测流程
- 用户只要评分提升建议 → 走评分提升模板
- 用户在做具体组件/页面的实施 → 走具体修复建议

### 3.2 5 铁律(违反任一不交付)

**铁律 1:空话黑名单**
- ❌ 禁止:专业 / 优雅 / 简洁 / 高级 / 精致 / 现代 / 好看 / 时尚 / 大气 / 质感
- ✅ 必须替换为:[具体特征]
- 示例:"高级" → "60%+ 留白 + 单一强调色 + 衬线 Display + 0 阴影"
- 检查:agent 输出前 grep 这 10 个词,命中则改写

**铁律 2:3 问适配性自检**
- 问题 1:这个哲学能解决用户的核心需求吗? → 答不出具体方法则不推荐
- 问题 2:用户的团队/技术栈能支撑吗? → 不能则不推荐(或标注"需 X 资源")
- 问题 3:有真实产品案例吗? → 没有则不推荐
- 任意 1 个否定 → 换方向

**铁律 3:Evidence 必填**
- 每方向至少 1 个真实产品案例(`references/philosophy-library.md` §6 中已映射)
- 引用格式:`[公司/项目] — [1 行说明]`
- 无 evidence = 不构成推荐,只是猜测
- 不知道的案例 → 标"待外部验证"占位

**铁律 4:互斥自检**(用 `references/philosophy-library.md` §2 末尾的兼容性矩阵)
- 3 个方向必须横跨至少 3 个不同流派
- ❌ 禁止同时推荐"极简 + 后现代"或"东方 + 极繁"(互斥)
- 🔶 部分兼容组合需标注"需谨慎"风险

**铁律 5:不堆叠无依据特征**
- 每个"关键视觉"必须能映射到真实产品案例
- 出现"应该有 X 特征"而无产品作证 → 删除该特征
- 反模式检测必须参考 `references/philosophy-library.md` §5 各流派反模式

### 3.3 报告类型与触发阶段对应

- 冷启动 → 强调"组件库适配" + "第一个可落地方向"
- 增量设计 → 强调"风格延展性" + "token 复用"
- 方向诊断 → 强调"归类" + "改进方向"(可以少给对比速览,多给诊断)
- 瓶颈突破 → 强调"差异最大化" + "风险标注"
- 标杆对齐 → 强调"路径" + "可行性" + "改造幅度"

### 3.4 简化模式

如果用户明确说"我就要这种风格"或"我只要 1 个方向"→ 跳过 3 方向推荐,直接给 1 个深度方向(仍应用 8 字段结构)。

### 3.5 视觉参考(可选字段)

每个方向建议可附带 1-2 张真实产品截图,展示该方向在业界产品中的实际应用。

**触发条件**:当前环境有用户授权且可用的外部搜索能力。

**搜索策略**:
- 搜索该哲学方向的代表性产品/公司(可参考 §6 真实产品案例库中的搜索关键词)
- 用当前环境可用的搜索或浏览能力查找
- 目视或读取页面内容,验证截图与该哲学方向匹配
- 每个方向最多 2 张截图

**输出格式**:
```
- 视觉参考:![描述](references/{company}-screen.png) *{公司名} — {1 行说明该产品如何体现此哲学方向} [{来源}]*
```

**降级处理**:外部访问不可用时,方向建议仅包含文字描述,不附带视觉参考。方向建议的有效性不依赖视觉参考。完整规则见 `references/integration.md`。

## 6. 评分提升建议文档

**生成时机**:审查报告输出并持久化后,立即生成。

### 6.1 模板

```markdown
# 评分提升建议

**项目**:[项目名]
**审查日期**:[YYYY-MM-DD]

## 评分总览

| 维度 | 当前 | 目标 | 差距 | 提升难度 |
|------|------|------|------|---------|
| 哲学一致性 | X/10 | Y/10 | ±N | 低/中/高 |
| 视觉层级 | X/10 | Y/10 | ±N | 低/中/高 |
| 细节执行 | X/10 | Y/10 | ±N | 低/中/高 |

**核心差距**:[1 句话描述当前与目标之间的核心差距]

1. **[提升步骤 1]** — Quick Win — 预计 5 分钟

   - 详见:review-report.md 行动层 Fix #N

2. **[提升步骤 2]** — 小修 — 预计 30 分钟
   - 操作:[具体操作]
   - 预期提升:+1.0 分
   - 依赖:步骤 1 完成后执行

3. **[提升步骤 N]** — 中修/大修
   - 操作:[具体操作]

   - 依赖:[依赖关系]

[同上结构]

### 6.2 生成规则

- 低于 8 分的维度:提供完整提升路径(当前评分 + 目标评分 + 步骤 + 预期提升)
- 8 分及以上的维度:仅提供维持建议(1-2 条注意事项)
- 提升路线图按优先级排序,标注依赖关系和预估工作量
- 引用审查报告时使用相对路径格式:`详见 review-report.md 行动层 Fix #N`

### 6.3 评分提升预估方法

- Quick Win(5 分钟内):+0.5 分
- 小修(< 30 分钟):+0.5~1.0 分
- 中修(< 2 小时):+1.0~2.0 分
- 大修(> 2 小时):+1.5~3.0 分
- 预估基于该维度的问题严重程度和修复覆盖面,非机械套用
- 同一维度多个步骤的累计提升不超过该维度的差距(目标 - 当前)

### 6.4 目标评分设定规则

- 目标评分 = 当前评分 + 合理提升空间
- 合理提升空间基于:问题数量、修复难度、修复覆盖面
- 目标评分不超过 9.5(10 分保留给完美无瑕的设计)
- 总体目标评分 = 各维度目标评分的算术平均

### 6.5 与审查报告的关系

- 评分提升建议文档是审查报告的"行动版"——从诊断到行动
- 每个提升步骤引用审查报告中的具体 Fix 条目
- 评分提升建议文档可独立阅读,无需同时打开审查报告
- 两份文档保存在同一目录下,便于对照

## 7. HTML 报告生成

> 基于 Markdown 报告内容,生成自包含 HTML 报告。借鉴 [The Unreasonable Effectiveness of HTML](https://thariqs.github.io/html-effectiveness/) 理念——用 HTML 的视觉层级、颜色编码和交互性替代 Markdown 文字墙。

### 7.1 全局 HTML 规范

1. **自包含单文件**:内联 CSS + 内联 JS,零外部依赖
2. **使用项目设计系统**:CSS 变量与 `index.html` 一致
   - 颜色:`--color-canvas-black` (#000) / `--color-arctic-white` (#fff) / `--color-subtle-gray` (#a9a9a9)
   - 字体:`--font-lausanne`(正文)/ `--font-victor-serif`(标题 h2)/ `--font-black-tie`(代码、标签、技术文本)
   - 间距:`--spacing-7` ~ `--spacing-64`
3. **纯黑白配色**:黑色画布、白色文字/边框、灰色辅助文字,无彩色
4. **零圆角**:`border-radius: 0`,1px 白色边框
5. **最大宽度 860px**,居中布局
6. **可折叠区块**:`<details>/<summary>`(无需 JS)
7. **跳转链接**:锚点 `#fix-N`、`#dimension-philosophy`

### 7.2 视觉元素规范

**评分条颜色规则**:
- 评分 ≥7:白色填充(`var(--color-arctic-white)`)
- 评分 <7:灰色填充(`var(--color-subtle-gray)`)

**严重度标签格式**:
- 白边框标签 + emoji:`P0 ⚠️` / `P1 ⚡` / `P2 💡`
- 无彩色背景,保持纯黑白

**交互式 SVG 雷达图**:
- 5 轴雷达图,与 `index.html` 雷达图风格一致
- 白色线条 + 半透明白色填充多边形
- 轴标签含维度名 + 分数,灰色文字

**颜色色块(Swatches)**:
- 硬编码颜色值用色块可视化:`<div style="background:#hex">` + hex 值 + 推荐 token
- 每个色块为方块(零圆角),1px 灰色边框

### 7.3 三层结构对应

- **结论层**:总体评分大字号 + 颜色编码(1-3 红/4-5 橙/6-7 黄/8-10 绿)、P0/P1/P2 色块标签、交互式 SVG 雷达图、Quick Win 高亮卡片
- **诊断层**:5 维度评分用进度条 + 颜色编码、专项检测总览用状态色块(✅绿/⚠️橙/❌红)、可折叠详情(`<details>/<summary>`)
- **行动层**:Fix 清单用严重度色块标签 + 工作量标签 + 跳转链接锚点、硬编码检测详情用颜色色块可视化、参考截图用缩略图 + 点击放大

### 7.4 视觉参考

- 审查报告:`case/Design Review — Landing Page.html`
- 评分提升建议:`case/Score Improvement — Landing Page.html`
- 设计建议报告:`case/Design Suggestion — Landing Page.html`

## 8. Demo 生成规范

> 当用户请求 Demo(触发词:demo/演示/看看效果/出个 html/做个示例/generate demo)或设计方向顾问输出后用户选定方向并要求可视化时,按以下规范生成单文件 HTML Demo。

### 8.1 触发条件

满足以下任一条件时生成 Demo:
1. 用户消息包含触发词:demo、演示、看看效果、出个 html、做个示例、generate demo、show me
2. 设计方向建议输出后,用户回复"方向 1/demo 方向 2/第一个/生成 demo"等
3. 用户明确说"给我一个可点击的版本"或"在浏览器里打开看看"

**默认不生成**(不主动生成以避免消耗),仅在用户请求时生成。

### 8.2 生成前确定

- **流派+哲学**:用户指定方向名 → 按方向生成;未指定 → 按设计建议报告中最推荐方向(方向 1)
- **页面类型**:默认品牌落地页;用户指定 Dashboard/表单/文档等 → 按 `references/philosophy-library.md` §11 页面模式库
- **梯度**:默认平衡版;用户说"保守点"/"激进点" → 切换梯度

### 8.3 技术规范

**强制要求**:
- **单文件 HTML**:所有 CSS 内联在 `<style>` 标签中,不依赖任何外部 CSS/JS/字体文件
- **系统字体兜底**:使用系统字体栈(按 §2 字体栈顺序,第一个放系统可用字体),不引入外部字体(确保离线可用)
- **图片占位**:使用 CSS 渐变/SVG pattern/纯色块/ASCII 字符占位,**不使用外部图片 URL**(避免失效)
- **响应式**:至少适配 3 个断点——桌面(1440px)、平板(768px)、移动端(375px)
- **内联 CSS 变量**:色板、字体、间距用 CSS 变量声明在 :root 中(符合 Token 思路)

**页面内容**(默认品牌落地页,用户指定页面类型时按 §11 模式库):
- 顶部 demo-note 条:标注流派+哲学名+该风格核心签名关键词
- Navigation(体现该流派导航特征)
- Hero 区域(体现该哲学**最强视觉签名**,这是判断 Demo 质量的核心)
- 3 个 Feature 卡片
- 1 个 CTA 区域
- Footer

### 8.4 生成执行流程

1. **解析请求**:确定流派+哲学名+页面类型(未指定时默认品牌落地页)
2. **读取素材**:从 `references/philosophy-library.md` §2 获取该哲学的色板+字体栈+核心特征;§5 获取反模式;§11 获取页面模式
3. **构建骨架**:从零按 §11 页面模式库 + §9 概念包(8 字段中的字段 7)构建结构
4. **注入 Token**:在 :root 中声明该哲学的 CSS 变量(色板/字体/间距/边框/阴影);必须使用 oklch() 值
5. **构建区块**:按字段 7b 布局范式+§11 模式库依次构建 Nav → Hero → Features → CTA → Footer
6. **添加交互**:为按钮/卡片/链接添加 hover 态 CSS(不写 JS,用 CSS transition 即可)
7. **响应式适配**:添加 @media 断点适配
8. **自检**:按 §8.5 清单逐项检查
9. **保存+通知**:写入文件,告知路径,建议用 `open [filepath]` 打开,提示可定制点

### 8.5 质量自检清单(生成后必须逐项检查)

- [ ] **哲学特征体现**:对照 §2 核心特征,至少 3 个该哲学的签名元素在 Demo 中可见(如野蛮 UI 必须有粗边框+硬阴影+零圆角)
- [ ] **色板准确**:使用 §2 中该哲学的 OKLch 色板(转为 CSS oklch() 值),不随意换色
- [ ] **无 AI slop**:没有无意义的装饰元素、没有 lorem ipsum(中文用中文,英文用合理英文)、没有紫色渐变
- [ ] **可交互**:所有按钮/链接/卡片有 hover 态样式
- [ ] **对比度**:正文文字与背景对比度 ≥ 4.5:1(WCAG AA)
- [ ] **响应式**:375px 宽度下内容不溢出、不横向滚动
- [ ] **无控制台错误**:HTML 结构正确,无未闭合标签
- [ ] **文件体积**:单文件 < 50KB

### 8.6 Demo 保存位置

生成的 Demo 保存到:`.monkren/demos/{philosophy-slug}-{page-type}-{YYYY-MM-DD}.html`

- kebab-case 命名,如:`mazzini-dashboard-2026-06-26.html`、`locomotive-landing-2026-06-26.html`
- 如果 `.monkren/demos/` 不存在则创建

### 8.7 生成后输出格式

```markdown
✅ Demo 已生成: [文件名]

**文件路径**: [完整路径]
**快速打开**: 在终端运行 `open [路径]` 即可在浏览器查看
