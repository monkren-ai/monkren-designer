# 设计改进

## 1. 建立修复基线

优先读取现有审查报告；没有报告时先执行 `../../04-review/5-dim-review/SKILL.md` 的只读检查。提取每条问题的位置、影响、证据、严重级别和验收标准。

## 2. 确认授权与范围

- 用户只要建议：输出修复路径，不改文件。
- 用户明确说修复、改进或实现：在给定项目范围内修改文件。
- 品牌、文案、信息架构等存在多种合理方向时，保留为决策项；不要用任意默认值覆盖真实产品判断。

## 3. 排序并实施

按 P0 → P1 → P2 排序。先处理结构、可用性和无障碍，再处理视觉一致性，最后做细节打磨。每项修复必须对应一个原始发现；避免借修复之名扩大功能范围。

当修复建议彼此冲突、需要大范围重构、会改变品牌核心或引入高成本实现时，先读取 `../../advisors/design-decision-council/SKILL.md` 获取只读第二意见。advisor 给出裁决后，本模块只实施用户已授权且结论明确的部分。

需要外部参考时，可使用当前环境可用的数据源或公开网页；不可用时继续基于本地设计系统和证据修复，不要求安装特定 agent 工具。

## 4. 验证

运行项目已有的构建、测试和 lint。视觉工具可用时检查关键桌面/移动视图与交互；不可用时执行静态检查并明确限制。修复后重跑相关审查项，确认没有引入对比度、focus、响应式或状态回归。

## 5. 交付

总结：修改文件、每项修复对应的发现、验证结果、仍未解决的判断项。需要单独报告时保存到 `.monkren/improvements/{project}-{YYYY-MM-DD}/report.md`。
