# Generate Variations：产出 3+ 个设计方案

为一个屏幕、组件或流程产出多个明显不同的设计变体，让用户能把最强的部分混搭。当用户要方案、替代、「不同 take」「看几个」时使用。**变体是通往好设计最便宜的路径**——一个设计是一次下注；三个让用户拒绝不想要的、组合想要的。

## Phase 1：确立基线

确认：**变化对象**（屏幕、组件、流程、视觉处理——范围决定有几个变体有意义）；**已有设计上下文**（变体应扎根于 UI kit / design system，除非明确要求挣脱）；**数量**（默认 3；5–6 是健康上限）；**轴偏好**（视觉、交互、布局、文案 / 语气——用户希望探索偏向哪边？）。

## Phase 2：选轴

挑 2–4 个维度来变化：视觉处理（色调、密度、阴影、圆角、字重）；布局（居中 vs 不对称、单栏 vs 多栏、满铺 vs 内嵌）；交互模型（单页 vs 多步、modal vs inline）；信息层级；语气；组件风格。写下每个变体在哪条轴上发力——这让对比对用户可读。

## Phase 3：带意图地构建——从稳妥到大胆

按从最循规蹈矩到最新颖的顺序：

1. **循规蹈矩。** 匹配已有模式与惯例——稳妥之选。
2. **精炼。** 同样结构，推一两个维度——更大胆的字体、更自信的布局、更有表现力的色彩。常常是用户真正的选择。
3. **新颖。** 一个真正不同的 take——非常规布局、强视觉隐喻、大胆审美。拉伸对话，露出用户自己都不知道的偏好。
4. **4–6（若要求）。** 沿光谱的混合，或另一条轴上的 wildcard。

**两头都要覆盖。** 全稳妥浪费用户时间；全狂野无视 brief。

## Phase 4：实质性变化，而非装饰性

变体应在布局、层级、什么是主、字体系统、密度、交互方式或文案策略上不同——而不只是 button 颜色、强调色色相或阴影透明度。如果两个太接近，砍掉一个换更实质的替代。用户应能用一句话说清任意两个变体的区别。

**构建前先把每个变体具体写下来**——明确的色板族、明确的字体搭配、明确的布局骨架，每个变体各写一份。多样性是设计出来的，不是盼出来的：不加约束的话，变体会漂向同一个默认长相（通常是那种暖色编辑 house style）。对新那个变体，刻意挑一个偏离分布、有意思的。

## Phase 5：放在单个文件里

默认输出一个自包含 HTML 文件：用语义 HTML、CSS Grid 和少量原生 JavaScript并排展示或切换全部变体。若目标项目已有框架，则在现有组件体系内实现同等结构。不要依赖仓库外的 starter，也不要产出散落的 `v1.html / v2.html / v3.html`。

## Phase 6：标注并推荐

每个变体用一两句话配文（「变体 2——精炼。同样结构，更有表现力的 headline 字体，更暖的色板。」）。配文是思考工具：如果你写不出清楚的一句，变体就还不够独立。

如果选定方向会造成重大产品承诺、高成本实现、难逆品牌或架构影响，先读取 `../../advisors/design-decision-council/SKILL.md` 获取只读第二意见。普通样式变体直接推荐，不增加流程。

结尾给一个明确推荐——用户拍板，但设计师要给意见（「我选变体 2——有 1 的稳妥，又多一点视觉自信」）。别含糊说所有方案一样好；并不是。

然后建议下一步：在选定方向上做一轮精修、在另一条轴上再来一轮变体、读取 `../../03-execute/make-a-prototype/SKILL.md` 验证交互，或在发布前读取 `../../04-review/polish-pass/SKILL.md`。
