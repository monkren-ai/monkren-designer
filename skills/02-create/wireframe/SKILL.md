# Wireframe：快速探索多种想法

产出低保真 wireframe 或故事板，在投入 hi-fi 设计之前探索某个流程、布局或想法。当用户想「探索方案」「画个草图」「看几个方向」时使用，或当问题足够开放、直接做 hi-fi 会浪费功夫时使用。**Wireframe 是一次性的**——它的价值在于方案的广度，而非其中任何一个的精细度。

## Phase 1：明确目标

确认：**探索的对象**（一个屏幕、一个流程、一种导航模式、一种信息层级、一种交互模型）；该屏幕或流程上的**用户目标**；**约束**（移动端或桌面端、已有上下文还是全新项目、不可妥协的元素）；**变体数量**（最少 3 个，每轮上限 5–6 个）；**变体轴**（布局？密度？步骤数？CTA 位置？）。

如果用户只说「画个注册流程的 wireframe」，主动提出 2–3 个轴（例如「单页表单 vs 多步向导 vs 渐进披露」），并询问要探索哪一个。

## Phase 2：确立 wireframe 规范

坚持 wireframe 的视觉语言，让用户把它读作 wireframe，而不是破损的 hi-fi：

- **纯灰度**——黑、白、2–3 种灰；不用 brand 色
- **系统无衬线字体**——没有字体个性；用户此时不应形成对字体的看法
- **内容区用方框**，带标签（「headline」「image」「feature card」）
- **图片用条纹 placeholder**，配等宽字体标签（`product shot 1200×800`）——绝不用真实图片；它们会抢走注意力
- **Ipsum 或骨架文案**——此阶段不用最终文案
- **欢迎标注**——在关键决策处加编号 callout

这是唯一允许手绘感 SVG（矩形、线条、简单图标）的场景——一切都处于同样的低保真层级。

## Phase 3：画出变体

产出**至少 3 个**在既定轴上有差异的变体。默认用一个自包含 HTML/SVG 文件并排放置单屏变体；流程变体在同一文件中做成每个 3–5 屏的小故事板。不要依赖未随仓库提供的 starter。

在布局（居中 / 分屏 / 网格）、信息密度、流程结构（单页 / 多步 / 渐进披露）、CTA 位置或导航模式上变化。按从最循规蹈矩到最新颖的顺序排列——当稳妥方案挨着冒险方案时，用户更容易挑出有意思的东西。

画之前先写下每个变体的差异化结构。不加约束的话，变体会收敛到几乎相同的布局——让差异是有意为之，并至少有一个真正偏离主流的。

## Phase 4：标注

每个变体加 2–4 个标注点，放在它旁边（不是单独一份文档），让用户把变体和它的理由一起读：

```
- 变体 1（单列向导）：简单、聚焦，但慢
- 变体 2（单页表单）：最快路径，第一印象重
- 变体 3（渐进披露）：兼顾两者，JS state 更多
```

## Phase 5：记录决策并交付

用户选定方向后，记录：选中的变体（或混合方案）、什么吸引了他们、他们明确拒绝了什么、过程中浮现的新约束。这会成为后续 hi-fi 工作的 brief。

然后建议读取 `../../03-execute/make-a-prototype/SKILL.md` 制作 hi-fi 可交互版本、把选中线框导入用户已有的演示工具，或再来一轮 lo-fi。不要在 wireframe 本身上投入 hi-fi 打磨——它们已经完成了使命。

总结：产出的变体、变体轴、建议的下一步、浮现的开放问题。
