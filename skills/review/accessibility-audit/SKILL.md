# 可访问性审查：WCAG 与包容性设计评审

审查当前设计的可访问性问题，覆盖对比度、语义结构、键盘导航、动效和表单。修复发现的所有问题。**良好的可访问性就是良好的设计——它惠及所有人。**

## Phase 1：确定审查范围

审查用户刚刚编辑或询问的 HTML 文件；否则审查最近修改的设计文件；如不明确，则询问。完整阅读，记录所使用的框架、目标级别（默认 WCAG AA）以及用户声明的任何约束。

## Phase 2：并行启动四个审查 agent

使用 ${AGENT_TOOL_NAME} 工具在一条消息中并发启动全部四个 agent，每个 agent 都带上完整文件内容。

明确指示每个 agent：报告发现的每一个问题，包括边缘情况和低严重度问题，并给出置信度和严重度估计。覆盖是 agent 的职责；过滤在聚合阶段（Phase 3）进行。

### Agent 1：对比度与色彩

1. **文本对比度。** 普通文本（小于 18px）需要 4.5:1；大文本（18px 及以上加粗，或 24px 及以上）需要 3:1；UI 组件（按钮、图标、focus ring）需要 3:1。对任何能解析出的颜色对计算实际比值；对每一个不达标的颜色对，标注其比值与所需的最低值。
2. **仅靠色彩传达状态。** 标记任何仅通过颜色传达的状态——没有图标的绿色/红色、没有下划线的链接、没有图例或标签的图表。
3. **易混淆组合。** 红+绿（最常见的色盲）、亮度相近的蓝+黄、白底浅灰、亮度相近的彩色底配彩色文字。
4. **纯白与纯黑。** 标记纯 `#FFFFFF` 配 `#000000`；推荐使用带微妙色调的（`#FAFAFA` / `#1A1A1A`）。这是风格建议，非 WCAG 要求。

### Agent 2：语义 HTML 与结构

1. **标题层级。** 恰好一个 `<h1>`，不跳级；标题描述内容，而非视觉大小。
2. **角色匹配元素。** 用 `<button>` 而非 `<div onclick>`；用 `<a href>` 而非样式化的 `<div>`；`<label for>` 关联到 input；使用 `<nav>`/`<main>`/`<article>`/`<section>`/`<aside>` 等 landmark。
3. **每张有意义的图片都要有 alt 文本。** 装饰性图片用 `alt=""`；有意义的图片描述其传达的内容（`alt="无线耳机，侧视图"` 而非 `alt="product"`）。
4. **表单 label。** 每个 input 都有关联的 `<label>`（或 `aria-label`）。仅靠 placeholder 文本不构成 label。
5. **仅在语义 HTML 无法满足时才用 ARIA。** 标记本可以用 `<button>` 却在 `<div>` 上加了 `role="button"` 的情况。

### Agent 3：键盘导航与 focus

1. **键盘可达。** 所有可点击元素都能通过 Tab 到达——仅 hover 的菜单、仅鼠标的下拉、键盘不可达的 modal 都不达标。
2. **合理的 tab order。** 遵循阅读顺序；标记大于 0 的 `tabindex` 值。
3. **交互模式。** modal 按 Escape 关闭；dropdown 用 Enter/Space 打开并用方向键导航；表单在字段内按 Enter 提交。
4. **可见的 focus ring。** 标记 `outline: none` 但没有提供与相邻背景达到 3:1 对比度的可见替代方案的情况。优先使用 `:focus-visible` 而非 `:focus`。
5. **跳转链接。** 在含有大量重复导航的页面上，建议添加"跳到主内容"链接。

### Agent 4：动效、表单与杂项

1. **尊重 `prefers-reduced-motion`。** 超过几百毫秒的动效需要 `@media (prefers-reduced-motion: reduce)` 块来缩短或移除动效。
2. **不得闪烁**超过每秒 3 次（光敏性癫痫风险）；标记并要求提供暂停控件。
3. **表单错误**要具体（"邮箱地址无效"而非"无效"），在视觉上并通过 `aria-describedby` 关联到对应字段。
4. **必填字段**用文字和/或图标加上 `required` 属性标记，而非仅靠颜色。
5. **input 类型与 autocomplete。** 使用 `type="email"`、`type="tel"`、`autocomplete` 属性以支持自动填充和移动键盘。
6. **点击区域**在触摸面上至少 44×44px。

## Phase 3：聚合并修复

等待全部四个 agent 完成。将它们的发现聚合为单一去重清单，并直接修复每个问题。对于边缘情况（如对比度为 4.4:1），仍然执行修复——可访问性是底线，不是上限。如果某个发现明显是误报或超出范围（例如 agent 标记了无法修改的第三方嵌入），则记录并跳过。

总结：按类别（对比度 / 语义 / 键盘 / 动效表单）分类的问题数、已修复的问题，以及留给用户处理的事项。
