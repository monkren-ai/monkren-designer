## Proposed Changes

### 改动 1:触发条件优化(philosophy.md §4 顶部重写)

**位置**:`references/philosophy.md` §4 设计方向顾问完整规则 — 触发条件小节

**新增 5 类触发场景识别**(agent 据此询问用户):

| 触发场景 | 关键信号 | 输出重点 |
|---------|---------|---------|
| 冷启动 (0→1) | "新项目,没 design system" | 强调入门成本、组件库适配 |
| 增量设计 (1→N) | "已有 system,想新增模块" | 强调风格延展性、token 复用 |
| 方向诊断 | "已有设计,想识别它属于什么" | 强调归类、特征对比、改进方向 |
| 瓶颈突破 | "陷同质化,想换风格" | 强调差异最大化、风险标注 |
| 标杆对齐 | "想对标 XX 产品/品牌" | 强调路径、改造幅度、可行性 |

**新增询问式确认**:
```
agent:"你的项目处于什么阶段?
A. 冷启动(0→1)
B. 增量设计(已有 system,加新模块)
C. 方向诊断(已有设计,识别流派)
D. 瓶颈突破(同质化,想换风格)
E. 标杆对齐(对标某个产品/品牌)
```

**Skip 条件保留并扩展**:
- 用户给了明确审查标准 → 走审查工作流
- 用户只要代码审查/硬编码检测 → 走检测流程
- 用户明确说"我就要这种风格" → 走设计建议的具体优化,不推荐 3 方向

### 改动 4:输出质量规则(philosophy.md §4 末尾新增)

**新增 §4.X 输出质量铁律**:

```markdown
### 输出质量铁律(违反任一不交付)

**铁律 1:空话黑名单**
- 禁止:专业/优雅/简洁/高级/精致/现代/好看/时尚/大气/质感
- 必须替换为:[具体特征] — 例:"高级" → "60%+ 留白 + 单一强调色 + 衬线 Display"
- 检查:agent 输出前 grep 这 9 个词,命中则改写

**铁律 2:3 问适配性自检**
- 问题 1:这个哲学能解决用户的核心需求吗? → 答不出具体方法则不推荐
- 问题 2:用户的团队/技术栈能支撑吗? → 不能则不推荐(或标注"需 X 资源")
- 问题 3:有真实产品案例吗? → 没有则不推荐
- 任意 1 个否定 → 换方向

**铁律 3:Evidence 必填**
- 每方向至少 1 个真实产品案例(§6 中已映射)
- 引用格式:[公司/项目] — [1 行说明]
- 无 evidence = 不构成推荐,只是猜测

**铁律 4:互斥自检(用现有兼容性矩阵)**
- 3 个方向必须横跨至少 3 个不同流派
- 不能同时推荐"极简 + 后现代"或"东方 + 极繁"(互斥)
- 部分兼容组合需标注"需谨慎"风险

**铁律 5:不堆叠无依据特征**
- 每个"关键视觉"必须能映射到真实产品案例
- 出现"应该有 X 特征"而无产品作证 → 删除该特征
```

## Assumptions & Decisions

| 假设 | 理由 |
|------|------|
| 40 哲学库内容不删改 | 之前的 `supplement-design-schools-philosophies.md` 已完成扩展,本次只做"应用层"优化 |
| 兼容性矩阵不重构 | 已完善(25+ 对关系) |
| 真实产品案例只列公认存在的产品 | 不编造,无依据则标"需验证" |
| HTML 报告视觉规范沿用 `case/Design Suggestion — Landing Page.html` 风格 | 保持设计语言一致 |
| 不修改 `case/` 中已有 HTML | 单独创建新版(若需要) |
| 不新增 7 派以外的流派 | 上次已决策 |
| SKILL.md 路由表不大幅重构 | 仅追加条目,保持稳定性 |
| 不引入外部数据/API | 仅基于已有哲学库 + 真实产品案例(手工整理) |

**关键决策**:
- **决策 1**:5 铁律(空话黑名单/3 问自检/Evidence 必填/互斥自检/不堆砌)写在 philosophy.md §4.X 而非独立章节 — 与原结构融合
- **决策 2**:§5/§6/§7 放在 §4 之后(用户接触顺序:先看规则,后看数据) — 符合 agent 工作流
- **决策 3**:3 方向横评对比表 — 保留为 markdown 表格而非 HTML(报告主体仍可读,HTML 版本在 implementation.md 模板中说明)
- **决策 4**:不强制"必须 Lazyweb 搜索真实产品案例" — 改为推荐,避免阻塞

## Risk & Mitigation

| 风险 | 影响 | 缓解 |
|------|------|------|
| philosophy.md 文件过大(新增 3 节后预计 2400+ 行) | 阅读/编辑成本上升 | 章节锚点 + References 路由表;暂不拆分(单文件易维护) |
| 真实产品案例库准确性 | agent 推荐基于错例 | 标"需 WebSearch 验证"的小众产品;不编造 |
| 7 × 10 矩阵的颗粒度 | 太粗无用,太细难维护 | 每个单元格简化为 ★/★★/★★★ 三档,够用即可 |
| 6 字段结构对简短场景过重 | 用户只想要 1 个方向时输出臃肿 | 触发条件新增"如果用户明确指定一个方向"→ 走简化模式 |
| 5 铁律过严可能阻碍 agent 输出 | 铁律 3 (Evidence) 在某些新兴领域难 | 允许"待验证"占位,但明确标注 |
| HTML 报告未同步更新 | 双格式不一致 | 实施后单独跑 1 个 HTML 生成验证 |
