# 项目能力评估计划

## 评估目标

对 monkren-design 项目进行全面能力评估，输出一份结构化的能力画像，帮助理解项目当前能做什么、做得有多好、以及哪些地方可以增强。

### 四、架构质量评估

| 维度 | 评级 | 说明 |
|------|------|------|
| 信息架构 | ★★★★★ | 三层四维骨架（信念层/标准层/执行层 × 主文档/深度参考），跨层引用规则清晰 |
| 模块化 | ★★★★☆ | 主文档 + 6 个参考文档 + 7 个子技能；子技能间有 Lazyweb 耦合 |
| 可维护性 | ★★★★☆ | 版本号统一（v1.2）、更新日期一致；部分内容跨文件重复定义 |
| Agent 兼容性 | ★★★★★ | 明确声明 Agent-Agnostic，支持 Claude Code/Cursor/Codex 等 |
| 降级韧性 | ★★★★★ | Lazyweb 3 级降级、事实验证降级、品牌提取失败处理 |

### 五、能力缺口识别

| 缺口 | 严重度 | 说明 |
|------|--------|------|
| 无 Android/Flutter/Kotlin 等效 SwiftLint 规则 | P1 ⚡ | 仅覆盖 Swift，Android 项目无法自动检测硬编码 |
| 无自动化测试/验证机制 | P1 ⚡ | 评分标准全靠 agent 自律，无客观校准手段 |
| 无设计系统组件库映射 | P2 💡 | 合规检查提到 PrimaryButton 等，但未映射到具体 DS（Ant Design/shadcn 等） |
| 缺少交互原型审查 | P2 💡 | 有交互体验审查维度，但缺少对原型/Figma 交互流程的审查规范 |
| 无多语言/国际化审查 | P2 💡 | 有中英文排版规范，但缺 i18n 审查检查项 |
| 缺少设计 token 自动提取工具 | P2 💡 | 品牌资产提取依赖手动 grep，无自动化脚本 |

### 六、与同类工具的差异化

| 差异化点 | monkren-design | 同类工具（如设计审查 SaaS） |
|----------|---------------|--------------------------|
| 哲学驱动 | 20 种设计哲学库 + 哲学一致性维度 | 通常只有视觉/功能维度 |
| 反 AI Slop | 8 类 slop 陷阱 + 文案/图标黑名单 | 通常无此维度 |
| 评分纪律 | 4 条铁律 + 自检迭代 | 通常无评分约束 |
| 代码级审查 | 硬编码检测 + SwiftLint + Token 架构 | 通常只审视觉 |
| Agent 原生 | 对话即审查，无需 UI | 需要 Web 界面 |
| 降级韧性 | 3 级降级 + 失败处理 | 通常依赖在线服务 |
| 品牌资产协议 | 5-10-2-8 原则 + 识别度排序 | 通常无此框架 |

## 实施步骤

1. **阅读所有核心文件**（已完成）—— SKILL.md + 6 个参考文档 + 7 个子技能
2. **梳理能力清单**——按功能域分类，标注来源和深度
3. **评估知识体系深度**——每个领域评级 + 说明
4. **评估架构质量**——信息架构/模块化/可维护性/兼容性/韧性
5. **识别能力缺口**——按严重度排序
6. **差异化分析**——与同类工具对比
7. **输出评估报告**——写入计划文件
