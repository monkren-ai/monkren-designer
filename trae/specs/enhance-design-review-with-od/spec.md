# 设计审查核心优化 Spec

## Why

当前 monkren-design 已完成从"设计创作+审查"到"纯设计审查"的定位转换，但审查能力仍有提升空间：审查报告仅文本输出缺乏视觉冲击力、评分纪律不够严格（容易评分通胀）、设计哲学库缺乏确定性色板/字体规格、品牌资产提取流程不够系统化、缺少结构化的审查前发现协议。Open Design 项目在审查可视化、评分纪律、确定性视觉方向、品牌资产协议、artifact lint 等方面提供了成熟的实践，值得吸收优化。

## What Changes

- **增强审查报告格式**：新增 SVG 雷达图可视化，使审查报告从纯文本升级为可视觉化呈现
- **强化评分纪律**：引入 OD 的评分铁律（不平均上浮、不评分通胀、必须引证、创新性允许低分）
- **升级设计哲学库**：为 20 种设计哲学补充确定性 OKLch 色板 + 字体栈规格，从抽象描述变为可执行规格
- **新增审查前发现协议**：结构化审查前问卷，锁定审查对象/范围/侧重点/设计上下文
- **新增品牌资产提取协议**：5 步品牌资产提取流程（定位→下载→提取色值→编写 brand-spec→确认）
- **新增 Artifact 结构检查**：HTML/CSS 结构性检查（断裂的 artifact 框架、缺失的必需文件、过期的 palette token）
- **新增设计系统参考基线**：引入 DESIGN.md 9 段式设计系统参考框架，作为审查时的对比基线
- **优化问题分级体系**：在现有 ⚠️/⚡/💡 基础上补充 P0/P1/P2 严重度分类，与 OD 的 checklist 文化对齐
- **新增自审迭代机制**：引入审查报告自检 + 迭代修正流程，确保报告质量

## Impact

- Affected specs: SKILL.md（多处增强）、references/critique-guide.md（评分纪律+报告格式升级）、references/design-styles.md（补充确定性规格）、references/design-context.md（品牌资产提取协议）、references/verification.md（artifact 结构检查）、references/workflow.md（发现协议+自审迭代）、references/content-guidelines.md（微调）
- Affected code: 无代码文件变更，全部为 Markdown 文档更新
- 新增文件：references/design-systems-baseline.md（设计系统参考基线）

## ADDED Requirements

### Requirement: SVG 雷达图审查报告

系统 SHALL 在审查报告中输出 SVG 雷达图，将 5 维度评分可视化呈现。

#### Scenario: 输出包含雷达图的审查报告
- **WHEN** 完成设计审查并输出报告
- **THEN** 报告包含内联 SVG 雷达图，5 个轴对应 5 个维度，填充区域直观展示评分分布；雷达图使用当前设计系统的 accent 色作为填充色，无设计系统时使用中性色

#### Scenario: 雷达图辅助快速定位薄弱维度
- **WHEN** 用户查看审查报告
- **THEN** 雷达图使薄弱维度一目了然（填充区域凹陷处），无需逐条阅读分数

### Requirement: 评分纪律铁律

系统 SHALL 严格遵守以下评分铁律，确保审查评分的客观性和区分度。

#### Scenario: 禁止评分通胀
- **WHEN** 审查者对所有维度打分均 ≥ 7
- **THEN** 系统应触发自检——7 分意味着"强"，如果所有维度都"强"则审查可能不够严格；至少 1 个维度应低于 7 除非设计确实卓越

#### Scenario: 禁止平均上浮
- **WHEN** 某维度在部分页面/组件表现优秀但在其他部分有明显问题
- **THEN** 评分取"最差持续段"而非平均值；如视觉层级在 hero 页优秀但 body 页混乱，应评 5-6 而非 7-8

#### Scenario: 评分必须引证
- **WHEN** 对某维度给出评分
- **THEN** 必须引用具体元素/文件/行号作为证据；"感觉不对"不可作为评分依据

#### Scenario: 创新性允许低分
- **WHEN** 审查生产交付物（非概念设计）
- **THEN** 创新性 5/10 是合理的，不应因"适当的保守"而惩罚功能性设计

### Requirement: 确定性设计哲学规格

系统 SHALL 为 20 种设计哲学补充确定性视觉规格，使每种哲学从"描述性参考"升级为"可执行规格"。

#### Scenario: 设计哲学包含确定性色板
- **WHEN** 用户选择或被推荐某设计哲学
- **THEN** 该哲学提供 OKLch 定义的确定性色板（主色 + 辅色 + 强调色 + 背景色 + 文字色），而非仅描述性文字

#### Scenario: 设计哲学包含确定性字体栈
- **WHEN** 用户选择或被推荐某设计哲学
- **THEN** 该哲学提供 display + body + mono 三级字体栈，含 fallback 链

#### Scenario: 设计哲学包含布局姿态提示
- **WHEN** 用户选择或被推荐某设计哲学
- **THEN** 该哲学提供布局姿态关键词（如"左对齐网格"、"居中对称"、"非对称破格"），指导布局方向

### Requirement: 审查前发现协议

系统 SHALL 在执行审查前通过结构化问卷锁定审查上下文，避免方向性误解。

#### Scenario: 审查前确认审查对象
- **WHEN** 用户发起审查请求
- **THEN** 系统确认：输入格式（截图/HTML/代码/Figma）、设计类型（web/App/幻灯片/信息图/落地页）、审查范围（整体/局部/单页/全流程）、审查侧重点（视觉/交互/代码合规/品牌一致性/全面）

#### Scenario: 审查前提取设计上下文
- **WHEN** 审查涉及具体项目
- **THEN** 系统按优先级提取：项目设计系统 → 品牌规范 → 现有代码库 → 通用设计原则基线

#### Scenario: 用户描述模糊时拒绝猜测
- **WHEN** 用户的审查请求描述模糊（如"帮我看看这个"）
- **THEN** 系统先问清楚再动手，不基于假设下结论

### Requirement: 品牌资产提取协议

系统 SHALL 在审查涉及具体品牌的设计时，执行 5 步品牌资产提取流程。

#### Scenario: 执行品牌资产提取
- **WHEN** 审查涉及具体品牌的设计产出
- **THEN** 系统执行：1) 定位品牌资产来源（官网/品牌指南/代码库）→ 2) 下载/读取资产文件 → 3) 提取色值（grep hex/rgb）→ 4) 编写 brand-spec 摘要 → 5) 向用户确认提取结果

#### Scenario: 品牌资产提取失败时
- **WHEN** 无法找到或提取品牌资产
- **THEN** 系统明确告知用户提取失败，不凭记忆猜测品牌色值，请求用户提供

### Requirement: Artifact 结构检查

系统 SHALL 对 HTML/CSS 产出执行结构性检查，作为审查的补充维度。

#### Scenario: 检查 HTML 结构完整性
- **WHEN** 审查 HTML 产出
- **THEN** 系统检查：断裂的标签、缺失的必需元素（doctype/html/head/body）、内联样式占比、未引用的 CSS 类

#### Scenario: 检查 CSS token 一致性
- **WHEN** 审查使用设计系统的代码
- **THEN** 系统检查：CSS 变量是否被定义但未使用、是否有硬编码值与 token 定义重复、palette token 是否过期（引用了已删除的变量名）

#### Scenario: 检查可访问性基线
- **WHEN** 审查任何 UI 产出
- **THEN** 系统检查：对比度（正文 vs 背景 ≥ 4.5:1）、可点击元素最小尺寸（44×44px）、图片 alt 文本、语义化标签使用

### Requirement: 设计系统参考基线

系统 SHALL 提供 DESIGN.md 9 段式设计系统参考框架，作为审查时的对比基线。

#### Scenario: 审查时引用设计系统基线
- **WHEN** 审查设计产出但项目无设计系统
- **THEN** 系统使用 DESIGN.md 9 段框架（色彩/排版/间距/布局/组件/动效/语调/品牌/反模式）作为结构化审查基线

#### Scenario: 审查时对比项目设计系统
- **WHEN** 审查设计产出且项目有设计系统
- **THEN** 系统将产出与项目设计系统逐段对比，标记偏离项

### Requirement: 问题严重度 P0/P1/P2 分类

系统 SHALL 在现有 ⚠️/⚡/💡 视觉标记基础上，补充 P0/P1/P2 功能性严重度分类。

#### Scenario: P0 阻断性问题
- **WHEN** 发现设计问题导致核心功能不可用或品牌识别度归零
- **THEN** 标记为 P0（对应 ⚠️致命），必须在发布前修复

#### Scenario: P1 重要问题
- **WHEN** 发现设计问题影响用户体验或设计一致性
- **THEN** 标记为 P1（对应 ⚡重要），应在下一迭代修复

#### Scenario: P2 优化建议
- **WHEN** 发现设计问题属于锦上添花
- **THEN** 标记为 P2（对应 💡优化），有空时改进

### Requirement: 自审迭代机制

系统 SHALL 在输出审查报告后执行自检，确保报告质量。

#### Scenario: 审查报告自检
- **WHEN** 完成审查报告输出
- **THEN** 系统执行自检：评分与描述是否一致、Fix 条目是否可操作、Quick Wins 是否真的 5 分钟可完成、是否有自相矛盾

#### Scenario: 自检发现问题时迭代修正
- **WHEN** 自检发现报告质量问题（如评分与描述不符、Fix 不可操作）
- **THEN** 修正报告后再输出，不交付有漏洞的审查报告

## MODIFIED Requirements

### Requirement: 审查报告输出格式

审查报告从纯文本格式升级为包含 SVG 雷达图的可视化格式。报告结构变更为：Header（审查对象+日期+审查员+1行结论）→ 雷达图（SVG）→ 5 维度卡片（各含评分+证据段落+Keep/Fix/Quick-win）→ 合并行动清单（Keep/Fix/Quick Wins）→ 硬编码值检测结果 → 设计系统合规性结果 → 品牌资产协议结果。

### Requirement: 5 维度评审体系

5 维度评审体系保持不变（哲学一致性/视觉层级/细节执行/功能性/创新性），但评分纪律增强：每维度评分必须附带 30-80 字证据段落，引用具体元素/类名/行号；禁止评分通胀和平均上浮；创新性维度允许低分。

### Requirement: 设计方向顾问

设计方向顾问保留 5 流派 × 4 方向结构，但每个方向从纯描述性升级为包含确定性 OKLch 色板 + 字体栈 + 布局姿态提示的可执行规格。

### Requirement: 工作流程

工作流程从 4 步升级为 6 步：1) 审查前发现（结构化问卷）→ 2) 提取设计上下文 → 3) 品牌资产提取（涉及品牌时）→ 4) 执行审查（5 维度 + 硬编码检测 + 合规性检查 + artifact 结构检查 + 可访问性检查）→ 5) 输出报告（含雷达图）→ 6) 自检迭代。
