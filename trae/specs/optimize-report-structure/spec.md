# 优化审查结果与设计建议报告文档 Spec

## Why

当前审查报告模板经过多轮能力扩展后，存在三重结构性问题：
1. **信息冗余**——5 维度评审卡片各自包含 Keep/Fix/Quick Win，合并行动清单又重复汇总，同一问题出现两次
2. **信息过载**——报告板块从最初的 5 个膨胀到 11+ 个（5 维度卡片 + 合并行动清单 + 硬编码 + 合规性 + 品牌资产 + 文案 + 图标 + UX + 流程），用户难以快速定位关键信息
3. **诊断与处方混杂**——评分/检测结果（"问题是什么"）和修复建议/设计方向（"该怎么改"）交织在同一板块，用户无法区分阅读

优化目标：让审查报告从"信息堆砌"升级为"分层递进的可操作文档"——先给结论，再给证据，最后给行动方案。

## What Changes

- **重构报告为三层递进结构**：结论层（1 分钟看懂）→ 诊断层（5 分钟理解问题）→ 行动层（按需深入修复）
- **消除 5 维度卡片与合并行动清单的重复**：维度卡片只保留评分+证据段落（诊断），Keep/Fix/Quick Win 统一收归行动层
- **合并专项检测结果为结构化表格**：硬编码/合规性/品牌资产/文案/图标/UX/流程 7 个独立板块合并为"专项检测总览"表格 + 按需展开详情
- **新增执行路线图**：在行动层增加优先级排序的修复路线图，标注依赖关系和预估工作量
- **新增设计建议报告格式**：当用户只需要设计方向建议时，提供轻量级的设计建议报告（区别于完整审查报告）
- **新增报告格式选择机制**：根据审查场景自动选择报告格式（完整报告 / 快速报告 / 设计建议报告）

## Impact

- Affected specs: SKILL.md（审查报告模板 + 报告格式选择）、references/implementation.md（报告模板深度展开 + 审查报告示例）
- Affected code: 无代码文件变更，全部为 Markdown 文档更新

## ADDED Requirements

### Requirement: 三层递进报告结构

审查报告 SHALL 按三层递进结构组织，从结论到证据到行动方案逐层深入。

#### Scenario: 结论层（1 分钟看懂）
- **WHEN** 用户打开审查报告
- **THEN** 第一眼看到：总体评分 + 评级 + 雷达图 + 1 句话结论 + P0 问题数 + 最关键的 1 条 Quick Win。结论层信息量控制在 5 行以内，无需滚动即可看完

#### Scenario: 诊断层（5 分钟理解问题）
- **WHEN** 用户需要理解具体问题
- **THEN** 诊断层提供：5 维度评分卡片（仅评分 + 证据段落，无 Keep/Fix/Quick Win）+ 专项检测总览表格。每个维度卡片控制在 3-5 行证据段落

#### Scenario: 行动层（按需深入修复）
- **WHEN** 用户准备动手修复
- **THEN** 行动层提供：合并行动清单（Keep + Fix + Quick Wins）+ 执行路线图 + 专项检测详情。行动层是报告的主体，按优先级和依赖关系组织

### Requirement: 消除维度卡片与行动清单的重复

5 维度评审卡片 SHALL 只保留评分和证据段落，Keep/Fix/Quick Win 统一收归合并行动清单。

#### Scenario: 维度卡片只含诊断信息
- **WHEN** 阅读 5 维度评审卡片
- **THEN** 每个维度卡片仅包含：维度名 + 评分 + 30-80 字证据段落。不包含 Keep/Fix/Quick Win

#### Scenario: 合并行动清单是唯一的行动指令来源
- **WHEN** 用户查找修复建议
- **THEN** 所有 Keep/Fix/Quick Win 只出现在合并行动清单中，不与维度卡片重复

### Requirement: 专项检测总览表格

7 个专项检测结果（硬编码/合规性/品牌资产/文案/图标/UX/流程）SHALL 合并为一个结构化总览表格，而非 7 个独立板块。

#### Scenario: 总览表格提供快速扫描
- **WHEN** 用户查看专项检测结果
- **THEN** 首先看到一个总览表格，每行一个检测维度，列包含：检测维度 / 状态（✅通过 / ⚠️N处问题 / ❌N处违规）/ 最严重的 1 条问题 / 关键修复建议。总览表格控制在 7 行以内

#### Scenario: 按需展开详情
- **WHEN** 用户需要某个检测维度的完整结果
- **THEN** 在总览表格下方，仅展开有问题的维度的详细结果。无问题的维度不展开详情

#### Scenario: 无问题时极简输出
- **WHEN** 所有专项检测均通过
- **THEN** 总览表格仅显示 7 行 ✅，无详情展开

### Requirement: 执行路线图

合并行动清单 SHALL 包含执行路线图，标注修复的优先级排序和依赖关系。

#### Scenario: 路线图按优先级排序
- **WHEN** 用户查看执行路线图
- **THEN** Fix 条目按 P0 → P1 → P2 排序，同优先级内按影响范围排序（全局问题优先于局部问题）

#### Scenario: 路线图标注依赖关系
- **WHEN** 某个 Fix 依赖另一个 Fix 先完成
- **THEN** 路线图标注依赖关系（如"依赖 #1 完成后执行"）

#### Scenario: 路线图标注预估工作量
- **WHEN** 用户查看 Fix 条目
- **THEN** 每个 Fix 标注预估工作量：Quick Win（5 分钟内）/ 小修（< 30 分钟）/ 中修（< 2 小时）/ 大修（> 2 小时）

#### Scenario: Quick Wins 独立为第一批次
- **WHEN** 执行路线图展示
- **THEN** Quick Wins 作为路线图的第一批次独立展示，标注"立即可做"

### Requirement: 设计建议报告格式

系统 SHALL 提供轻量级的设计建议报告格式，当用户只需要设计方向建议时使用。

#### Scenario: 用户请求设计建议而非完整审查
- **WHEN** 用户请求"设计建议""设计方向""如何改进"而非"审查""评审"
- **THEN** 系统输出设计建议报告，而非完整审查报告

#### Scenario: 设计建议报告结构
- **WHEN** 输出设计建议报告
- **THEN** 报告包含：当前状态评估（2-3 句话）+ 3 个设计方向建议（每个含方向名/视觉特征/适用场景/关键改动）+ 优先级排序 + Quick Win

#### Scenario: 设计建议报告与审查报告的关系
- **WHEN** 用户先获得设计建议报告，后请求完整审查
- **THEN** 完整审查报告可引用设计建议报告中的方向作为哲学一致性评审的参照

### Requirement: 报告格式选择机制

系统 SHALL 根据审查场景自动选择报告格式，并在审查前发现阶段确认。

#### Scenario: 完整报告
- **WHEN** 用户请求全面审查，或审查对象复杂（落地页/App UI/品牌设计）
- **THEN** 使用三层递进完整报告格式

#### Scenario: 快速报告
- **WHEN** 用户请求快速审查，或审查对象简单（单个组件/图标/局部修改）
- **THEN** 使用快速报告格式：仅包含结论层 + 行动层（无诊断层的维度卡片，无专项检测详情）

#### Scenario: 设计建议报告
- **WHEN** 用户请求设计方向建议而非审查
- **THEN** 使用设计建议报告格式

#### Scenario: 格式确认
- **WHEN** 审查前发现阶段
- **THEN** 在现有问卷中增加"报告格式"选项：完整报告 / 快速报告 / 设计建议报告，默认根据审查范围自动选择

## MODIFIED Requirements

### Requirement: 审查报告模板

审查报告模板从"平铺式"重构为"三层递进式"。

旧结构（11+ 板块平铺）：
1. Header → 2. 雷达图 → 3. 5维度卡片（各含Keep/Fix/Quick Win）→ 4. 合并行动清单 → 5. 硬编码 → 6. 合规性 → 7. 品牌资产 → 8. 文案 → 9. 图标 → 10. UX → 11. 流程

新结构（三层递进）：
1. **结论层**：Header + 雷达图 + 1 句话结论 + P0 数 + 关键 Quick Win
2. **诊断层**：5 维度评分卡片（仅评分+证据）+ 专项检测总览表格
3. **行动层**：合并行动清单（Keep + Fix + Quick Wins）+ 执行路线图 + 专项检测详情（仅展开有问题的）

### Requirement: 5 维度评审卡片格式

5 维度评审卡片从"评分 + 证据 + Keep/Fix/Quick Win"精简为"评分 + 证据段落"。Keep/Fix/Quick Win 统一收归合并行动清单。

旧格式：
```
#### 哲学一致性：7/10
[证据段落]
**Keep**：[做得好的具体方面]
**Fix**：[需要改进的具体方面]
**Quick Win**：[5 分钟可完成的改进]
```

新格式：
```
#### 哲学一致性：7/10
[30-80 字证据段落，引用具体元素/类名/行号]
```

### Requirement: 审查前发现问卷

审查前发现问卷在现有 4 项确认基础上增加第 5 项"报告格式"。

新增确认项：

| 确认项 | 选项 | 必要性 |
|--------|------|--------|
| 报告格式 | 完整报告 / 快速报告 / 设计建议报告 | 可选（默认根据审查范围自动选择：整体→完整，局部→快速，方向建议→设计建议） |

### Requirement: 审查报告示例

references/implementation.md 中的 3 个审查报告示例需按新三层递进结构重写。
