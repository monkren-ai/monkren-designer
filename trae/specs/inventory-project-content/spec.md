# 梳理项目内容 Spec

## Why
monkren-design 项目经过多轮迭代，内容量已相当庞大（主文档 550+ 行、6 个 references 文件共 3000+ 行、7 个子 skill），需要系统性梳理当前内容结构，识别冗余、缺失和可优化的部分，为后续迭代提供清晰的基线。

## What Changes
- 梳理并文档化项目当前内容结构（主文档 + references + 子 skill 的层级关系）
- 识别各文档间的内容重叠与冗余
- 识别内容缺口（该有但缺失的规范）
- 标记内容版本不一致处
- 输出项目内容地图（content map），作为后续优化的参照基线

## Impact
- Affected specs: 无直接影响，此为纯梳理任务
- Affected code: 无代码变更，仅产出内容分析文档

## ADDED Requirements

### Requirement: 项目内容地图
系统应产出一份结构化的项目内容地图，覆盖以下维度：

#### Scenario: 完整内容梳理
- **WHEN** 执行项目内容梳理
- **THEN** 产出包含以下内容的内容地图：
  1. 主文档 SKILL.md 的三层架构内容索引（信念层/标准层/执行层各包含哪些模块）
  2. 6 个 references 文件的内容索引（每个文件覆盖哪些主题、与主文档的映射关系）
  3. 7 个子 skill 的功能定位索引（每个 skill 的触发场景、核心能力、与主 skill 的关系）
  4. 内容重叠矩阵（哪些主题在多个文件中重复出现、重复程度如何）
  5. 内容缺口清单（审查工作流中引用但未展开的规范、用户可能需要但未提供的指引）
  6. 版本不一致标记（各文件标注的版本号和更新日期是否一致）

### Requirement: 内容重叠分析
系统应识别并标记以下类型的重叠：

#### Scenario: 跨文档内容重叠
- **WHEN** 同一主题在多个文件中出现
- **THEN** 标记重叠类型：
  - **合理引用**：主文档一句话引用 + 指向 references 深度展开（三层边界规则要求）
  - **冗余重复**：多个文件以相似深度展开同一内容，无明确主体层归属
  - **碎片化**：同一主题分散在多个文件中，缺乏统一入口

### Requirement: 内容缺口分析
系统应识别以下类型的缺口：

#### Scenario: 审查工作流引用但未展开
- **WHEN** 主文档或 implementation.md 引用了某个规范但 references 中无对应深度内容
- **THEN** 标记为内容缺口

#### Scenario: 用户常见需求但无对应指引
- **WHEN** 用户可能提出的需求在当前内容体系中无对应模块
- **THEN** 标记为潜在内容缺口

### Requirement: 子 skill 生态梳理
系统应梳理 7 个子 skill 与主 skill 的关系：

#### Scenario: 子 skill 定位分析
- **WHEN** 执行子 skill 生态梳理
- **THEN** 产出每个子 skill 的：
  - 功能定位（解决什么问题）
  - 触发场景（什么时候用）
  - 与主 skill 的关系（互补/增强/独立）
  - 依赖关系（是否依赖 Lazyweb MCP）
  - 与其他子 skill 的边界（什么场景用 A 而非 B）
